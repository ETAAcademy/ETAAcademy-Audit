# RowHammer and Beyond: The Evolution of Hardware Security Attacks from Memory, Power, to Processor Architectures

Modern hardware security is increasingly threatened by vulnerabilities that exploit the physical and architectural properties of components rather than software logic flaws. RowHammer represents a foundational DRAM disturbance error where rapid toggling of memory rows induces bit flips in adjacent physical locations, effectively breaking memory isolation and enabling privilege escalation.

Building upon this, Flip Feng Shui (FFS) weaponizes operating system features like memory deduplication to deterministically position victim data into vulnerable memory regions, allowing for precise attacks against cryptographic keys such as RSA moduli. Power management mechanisms like Dynamic Voltage and Frequency Scaling (DVFS) further expand the attack surface, as demonstrated by CLKSCREW, which manipulates voltage and frequency via software to induce timing violations and extract secrets from hardware-isolated environments like ARM TrustZone. Similarly, Hertzbleed reveals that data-dependent power consumption in modern processors can be remotely observed through frequency variations, invalidating the core assumptions of constant-time programming and enabling remote timing attacks against post-quantum cryptographic algorithms like SIKE.

Collectively, these vulnerabilities highlight a critical need for a security-centric redesign of hardware architectures and a reevaluation of the trust boundaries between software and the underlying physical infrastructure.

## 1. RowHammer: Exploiting DRAM Disturbance Errors

**RowHammer** is a disturbance error affecting various memory technologies, most notably DRAM. The vulnerability quickly became one of the most well-known hardware security flaws (e.g., CVE-2014-3609 and subsequent variants) and prompted a shift toward more robust system-level approaches for ensuring memory reliability and security.

RowHammer exploits disturbance errors in DRAM. The attack method is remarkably simple: an attacker repeatedly and rapidly accesses the same DRAM rows, such as Row A and Row B, in an alternating pattern (Row A → Row B → Row A → Row B ...). This behavior is known as **“hammering.”** As a result, the repeated activations disturb the electrical charge stored in physically adjacent rows—referred to as **victim rows**—which may cause **bit flips**.

A **disturbance error** is a class of hardware failure in which an operation on one component unintentionally affects nearby components. Similar effects have been observed in multiple storage technologies, including SRAM (read/write disturbance), Flash memory (program/read disturbance), and hard drives (adjacent track interference). In the case of DRAM, RowHammer demonstrates that repeated memory access can corrupt data stored in nearby rows.

The reason RowHammer became a critical **security vulnerability** is that an attacker can flip bits in memory locations that they do not own. For instance, while the attacker can only access user-level memory, the induced bit flips may occur in sensitive regions such as **page tables or kernel memory**. By corrupting page table entries, an attacker may gain unauthorized privileges and ultimately achieve **privilege escalation**. This phenomenon effectively **breaks memory protection**, a fundamental assumption in traditional computer security models.

Conventional security models assume that **memory isolation implies reliability**: user processes cannot modify the memory of other processes without explicit permission. RowHammer breaks this assumption because memory corruption occurs without direct write access; instead, it is triggered solely through repeated memory accesses. For this reason, RowHammer is widely regarded as a **hardware-level security vulnerability**.

Initially, it was believed that DRAM manufacturers could resolve the issue through improved chip design. However, subsequent research revealed that newer DRAM technologies remain susceptible to RowHammer. In fact, the problem tends to worsen as semiconductor process nodes shrink, because the physical spacing between memory cells becomes smaller and electrical interference becomes more pronounced.

Consequently, researchers began proposing **system-level defenses**, where reliability and security are ensured not only by hardware manufacturers but also by the system architecture and operating system. Representative solutions include:

- **Target Row Refresh (TRR)** implemented in memory controllers
- **Operating-system-level physical memory isolation**, such as page coloring
- **Software-based hammer pattern detection**

These approaches aim to mitigate disturbance errors even when the underlying hardware remains vulnerable.

A key empirical observation is that repeated activation of a single DRAM row can trigger bit flips in adjacent rows once the number of activations exceeds a certain threshold. Experimental studies report a **critical activation count** of approximately $N_{act} \approx 139{,}000$.

This means that within a single refresh interval (typically **64 ms**), roughly 139,000 row activations can corrupt data in neighboring rows. This threshold is surprisingly low, meaning that a simple user-level program executing a tight memory-access loop can easily exceed it, making the vulnerability practical to exploit.

Physically, the disturbance effect is believed to be caused by several mechanisms, including:

- **Electromagnetic coupling (crosstalk)** between adjacent memory cells
- **Charge leakage** in DRAM capacitors
- **Hot-carrier injection or electrical bridging**

DRAM stores information as charge in tiny capacitors, and this charge naturally dissipates over time. To maintain data integrity, DRAM cells must be periodically **refreshed**. If repeated row activations cause sufficient disturbance before the next refresh occurs, the stored charge in adjacent cells may drop below the noise margin, resulting in a bit flip.

Although DRAM manufacturers have long performed hammer-style stress tests during chip production, RowHammer errors continue to appear in commodity hardware under real-world conditions. As technology scaling continues, purely circuit-level isolation becomes increasingly difficult to maintain.

To address this challenge, researchers have proposed practical mitigation techniques such as **PARA (Probabilistic Adjacent Row Activation)**. PARA provides a lightweight mechanism for CPU and memory controller designers to probabilistically refresh neighboring rows during frequent activations, significantly reducing the likelihood of disturbance-induced errors.

### DRAM Architecture Basics

DRAM consists of ranks, banks, and rows. Data is maintained through periodic refreshes and accessed by "opening" a row into a row buffer via wordline activation. **DRAM Cell**: the basic unit of memory, consisting of one capacitor (storage) and one access transistor (switch).

- **Wordlines and Bitlines**: a grid of wires where wordlines control the access transistors of a row, and bitlines transfer charge to the row buffer.
- **Sense Amplifier (Row Buffer)**: a circuit that detects weak charge in a cell, "amplifies" it to full voltage (0 or 1), and then writes it back to the cell (restoring data).
- **Destructive Read**: the process of reading charge from a DRAM cell into the row buffer destroys the original charge in the cell, requiring an immediate write-back. **Timing Constraints**: hardware-enforced command delays used to allow electronic components (like sense amplifiers) to stabilize.

The hierarchical structure of DRAM goes from the upper layers (Ranks and modules) down to the lower layers (Banks, rows, and cells). A DRAM Rank consists of multiple chips working together to provide a wide data bus (usually 64 bits). Each chip is divided into several Banks that share a row buffer. To access data, a row must be "opened" (ACTIVATE command), which raises the wordline voltage and transfers data to the row buffer. After reading or writing data, the row must be "closed" (PRECHARGE command) to clear the bitlines and row buffer for the next access. Since capacitors leak charge, every cell must be "refreshed" within its retention time (guaranteed 64 ms for DDR3) using the REFRESH command, which internally performs ACTIVATE and PRECHARGE cycles.

Timing constraints: row cycle time $t_{RC} \approx 50 \text{ ns}$ is a hardware parameter defining the frequency limit of the RowHammer effect—the minimum time interval between two ACT commands to the same row. It limits the maximum frequency of wordline toggling. The maximum possible "hammer" (activations) count within a refresh window is limited by $64 \text{ ms} / t_{RC}$. If $t_{RC}$ is 50ns, one could theoretically activate a row $1.28 \times 10^6$ times within a 64ms window, far above the 139K threshold identified in Section 1. $T_{retention}$ (retention time) is the maximum time a cell can hold its charge without refreshing. $N_{ref}$ (refresh commands) is 8,192 commands per 64 ms window. $t_{REFI}$ (refresh interval) is the average time between REF commands (7.8 microseconds). The maximum activations per refresh window ( $N_{max} = \frac{T_{refresh\_window}}{t_{RC}}$ ), where $T_{refresh\_window}$ is the standard 64 ms DDR3 window. This represents the total number of times any row in a bank can be toggled before a specific row in that bank is guaranteed to be refreshed. This calculation confirms that the "RowHammer" vulnerability is physically possible under standard DRAM timing constraints.

**Row Buffer Locality**: all accesses to an "open" row are handled by the row buffer. This means subsequent column reads to the same row do not toggle the wordline, explaining why simple sequential reads to a row do not cause errors. "Refresh" is electrically identical to an "Activate" followed by a "Precharge." This is key to understanding how to mitigate errors (by refreshing adjacent rows). Multiple banks allow concurrent access, increasing "hammering" efficiency by alternating accesses between different rows.

### Mechanisms of Disturbance Errors

Disturbance errors are caused by unintended electrical interactions between wordlines and adjacent cells, where repeated voltage switching accelerates charge leakage beyond the capacitor's ability to maintain data between refreshes. Physical mechanisms for device failure include noise injection, bridging, hot-carrier injection, and cumulative interference.

- **Noise Injection**: a process where a changing signal on one wire (aggressor) induces a voltage or current on a nearby wire (victim) through capacitive or inductive coupling.
- **Bridging**: a physical defect or "short circuit" where a conductive path exists between two points that should be isolated.
- **Hot-Carrier Injection (HCI)**: a phenomenon in solid-state devices where electrons or holes gain enough kinetic energy to overcome potential barriers and become trapped in the gate dielectric or injected into adjacent regions, altering device characteristics.
- **Cumulative Interference**: many tiny transient disturbances accumulate over time, leading to a permanent or semi-permanent state change (bit flip).

While the exact device-level cause cannot be confirmed without internal DRAM design access, three main disturbance mechanisms can be hypothesized. First, toggling a wordline might inject noise into adjacent wordlines, causing their transistors to partially turn on and allow charge leakage. Second, wordline toggling could stress existing "bridges" (manufacturing defects), accelerating charge transfer. Third, repeated toggling might cause hot-carrier injection, altering the threshold voltage of adjacent access transistors or directly changing cell characteristics. The key point is that these effects are cumulative; a single activation is harmless, but hundreds of thousands of activations within a short time (before refresh) are sufficient to deplete the charge in a capacitor, leading to a bit flip.

### Real-world RowHammer Exploits

User-level programs can use refresh instructions to bypass the CPU cache and repeatedly toggle DRAM wordlines, inducing bit flips in unprivileged memory and proving that RowHammer is a practical security threat. System programming and architectural concepts:

- **CPU Cache Line**: a small block of memory (usually 64 bytes) in the CPU cache. Accessing cached data does not trigger a DRAM access.
- **Cache Flush (`clflush`)**: an x86 instruction used to invalidate a specific cache line, forcing the next access to that address to go directly to DRAM.
- **Memory Fence (`mfence`)**: an instruction used to ensure all previous memory operations (like flushes) are completed before continuing, preventing CPU out-of-order execution from optimizing away the flush.
- **Bank Mapping**: the internal logic used by the memory controller to map physical addresses to specific banks and rows in DRAM.
- **Address Translation**: the OS mechanism (paging) used to map virtual addresses to physical addresses. To attack a specific row, an attacker must understand or bypass this mapping.

A minimal assembly program (Code 1a) can induce bit flips. This program reads data from two different addresses (X and Y) that map to the same bank but different rows. By alternating reads and using `clflush`, it forces the memory controller to close row X and open row Y (and vice versa) in a continuous loop. This was verified on Intel (Sandy Bridge, Ivy Bridge, Haswell) and AMD (Piledriver) systems. A control experiment (Code 1b) reads data from only one address and does not cause any errors because that row remains open in the row buffer, preventing wordline toggling. This proves that the disturbance is caused by the _activation_ (opening/closing) of rows, not just data transfer.

**Code Snippet 1a (Causes Errors)**: the `mov` instructions (lines 2-3) load data from memory into registers. The `clflush` instructions (lines 4-5) remove addresses X and Y from all levels of the CPU cache. The `mfence` instruction (line 6) acts as a serialization point to ensure flushes are committed. The `jmp` instruction (line 7) creates an infinite loop. Since X and Y map to different rows in the same bank, the memory controller must issue an `ACTIVATE` for X, then a `PRECHARGE`, then an `ACTIVATE` for Y. This achieves an **(Open-Read-Close)**$^N$ access pattern, maximizing the wordline toggling frequency.

### Characterizing Errors with FPGA

To systematically characterize disturbance errors, an FPGA-based platform was used, providing cycle-accurate control over DRAM commands and ambient temperature for 129 different memory modules. **FPGA (Field Programmable Gate Array)** is a programmable semiconductor device used to simulate specific hardware logic, here used to build a custom memory controller. **Cycle-Accurate Control**, the ability to issue commands (ACT, PRE, REF) at specific clock cycles, bypassing commercial CPU optimizations and abstractions. **Activation Interval ( $AI$ )** is the time interval between consecutive activations of a row during testing. **Refresh Interval ( $RI$ )** is the time interval between consecutive refreshes of the DRAM module during testing. **Data Pattern ( $DP$ )** is a specific arrangement of 0s and 1s written to DRAM to test pattern-dependent sensitivity.

A testbed was built using eight Xilinx FPGA boards, each equipped with a custom DDR3-800 controller. This setup provided raw access to DRAM, avoiding the data scrambling typically performed by commercial CPUs. 129 modules from manufacturers A, B, and C were tested, covering a wide range of production dates (2008-2014). Two types of tests were used: `TestBulk`, to identify all vulnerable cells in a module; and `TestEach`, to identify which aggressor row causes which victim cell to flip. Tests typically ran at 50°C with $AI = 55 \text{ ns}$ and $RI = 64 \text{ ms}$.

Activation count ( $N$ ): the number of times a row is toggled in a test is defined as $N = \frac{2 \times RI}{AI}$, where the total number of hammer events equals the test duration divided by the toggling frequency. The factor of 2 ensures that for any given row, the toggling duration completely overlaps with at least one refresh interval, regardless of internal refresh alignment. For default values, $N \approx 2.33 \times 10^6$.

- $N$: total toggles per test, the "hammer" intensity
- $RI$: Refresh Interval, default 64 ms
- $AI$: Activation Interval, default 55 ns

**2a: TestBulk(AI, RI, DP)**: initializes parameters and writes the test pattern to the entire module. It iterates through every row ( $r$ ) in the module. For each row, it performs $N$ activation/read/precharge cycles (lines 8-11). After all rows are tested, it reads the entire module to find flipped bits. This test identifies the union of all cells vulnerable to interference from _any_ row in the module.

### PARA: A Probabilistic Solution

PARA is a stateless probabilistic mechanism that mitigates RowHammer attacks by refreshing adjacent rows with a very low probability upon every row closure, providing high reliability with negligible performance overhead. **Error Correcting Code (ECC)** is a hardware capable of detecting and correcting bit flips, though typically only 1 bit per 64-bit word. **Stateless Mechanism**: does not require tracking past events (e.g., counting activations), making it simpler and cheaper to implement in hardware. **PARA (Probabilistic Adjacent Row Activation)** decides whether to refresh adjacent rows via a "biased coin flip." **Binomial Distribution** uses a discrete probability distribution to model the probability that a row is _not_ refreshed even after multiple "hammer" events.

Most schemes were excluded due to excessive overhead (e.g., increasing the refresh frequency of all rows by 8x) or complexity (e.g., using counters to track "hot" rows). PARA refreshes adjacent rows with probability $p$ upon every row closure. Being probabilistic, PARA is stateless and requires no additional storage. With $p = 0.001$, the probability of an error occurring within a year is extremely low ( $9.4 \times 10^{-14}$ for 100K hammers), while performance degradation is less than 0.2%.

PARA Error Probability Model: an error occurs in a victim row if it is never refreshed during $N$ activations of its neighbor. For a victim row to fail, every "biased coin flip" during $N_{th}$ hammer events must be "tails" (no refresh). As $N_{th}$ increases (increasing hammer intensity), the probability of _never_ refreshing a neighbor drops exponentially. This fits the "N-body" nature of the problem; the more you hammer, the more "chances" PARA has to save the corrupted row.

$$
P_{\text{error}} = (1 - p/2)^{N_{th}}
$$

- $P_{\text{error}}$: failure probability per refresh window, must be minimized
- $p$: probability of triggering PARA, typically 0.001
- $p/2$: probability of refreshing a _specific_ neighbor, assuming 2 neighbors
- $N_{th}$: activation threshold, approx. 139,000

Annual Error Probability: this value scales the single-window error probability to a full year of continuous refreshes. It shows that even a probabilistic solution can provide "statistical certainty" far exceeding the reliability of other hardware (like hard drives).

$$
P_{\text{year}} = 1 - (1 - P_{\text{error}})^{T_{\text{year}}/T_{\text{window}}}
$$

- $T_{\text{year}}$: seconds in a year, $3.15 \times 10^7$ s
- $T_{\text{window}}$: refresh window, 64 ms

PARA requires the memory controller to know which rows are physically adjacent. Manufacturers should disclose this "logical-to-physical" mapping in the SPD (Serial Presence Detect) ROM. If a row is remapped due to defects, it might have more than two neighbors. Higher $p$ values (e.g., $0.005$) and a wider pool of candidate rows are suggested to address this. PARA's main competitor is the "hot row counter." Counters are computationally expensive ( $O(N_{\text{rows}})$ ), and approximate counters (Bloom filters) are prone to collisions, leading to unnecessary refreshes. The average performance loss is only 0.197%, which is very low compared to the 20-30% overhead required for a deterministic 8x refresh rate. **Targeted Refresh** commands added to subsequent DDR standards essentially implement the hardware side of PARA or similar adjacent refresh logic.

## 2. Flip Feng Shui (FFS): Bypassing Isolation via Deduplication

Flip Feng Shui (FFS) is a novel attack vector that uses memory deduplication as a memory manipulation primitive, combined with the Rowhammer hardware vulnerability, to achieve targeted, controllable bit flips in arbitrary physical memory across the software stack, completely bypassing the need for software vulnerabilities. **Rowhammer** repeatedly reads to memory rows cause charge leakage, inducing bit flips in adjacent rows. **Memory Deduplication (KSM in Linux)** is an OS feature that merges identical physical memory pages into a single shared page to save memory. **Memory Manipulation** (e.g., via deduplication) forces victim data into attacker-predictable, controllable physical memory locations known to be vulnerable to Rowhammer bit flips.

Traditionally, hardware failures like bit flips were viewed as reliability issues or limited side-channel attacks. By combining physical memory manipulation primitives (exploiting Linux KSM deduplication) and hardware vulnerabilities (Rowhammer), an attacker can perform targeted memory corruption on co-resident virtual machines. An attacker creates a memory page containing known target content, waits for the OS to deduplicate and map it to the same physical page as the victim, and then uses Rowhammer to induce a bit flip. This method was used to breach OpenSSH public keys and Debian/Ubuntu update mechanisms on co-resident victim VMs, achieving full control without exploiting any software vulnerabilities. The attack assumes the attacker can accurately predict the content of target pages on the victim VM, which is feasible for public keys or standard config files but harder for dynamic data. It also relies on the OS having memory deduplication enabled—common in cloud environments for optimization, but it can be disabled as a primary mitigation.

Unlike previous probabilistic or local Rowhammer exploit methods, Flip Feng Shui provides a fully controllable, deterministic bit-flip primitive that works across VM boundaries without relying on software bugs, prompting an immediate re-evaluation of hardware trust in cloud security. **Heap/Memory Spraying (Software)**: existing techniques (like Heap Feng Shui) used to organize virtual memory predictably for buffer overflow exploits. FFS extends this concept to physical memory. **Side Channels**: unintended information leakage (e.g., via the timing of memory deduplication), previously used mainly for information disclosure rather than active memory corruption.

Early Rowhammer exploits (like Seaborn's) either required local execution (escaping sandboxes) or relied on probabilistic memory reuse during system crashes. Other exploits required direct access to page mappings or exploited JavaScript cache eviction mechanisms. FFS's fundamental difference is that it is the first technique to achieve reliable and deterministic control over _arbitrary_ physical memory across almost completely isolated boundaries (like co-hosted VMs) without any pre-existing software bugs. It also elevates memory deduplication from a passive information-leakage side channel to an active offensive weapon.

Hardware bit flips can no longer be dismissed as production reliability issues or probabilistic edge cases. By combining memory manipulation and templating techniques, FFS turns hardware unreliability into a precise cyber-attack weapon. The systems security community should immediately update threat models to account for deterministic physical memory corruption.

Flip Feng Shui _synthesizes_ two known, benign concepts (Rowhammer as a hardware flaw + deduplication as an OS feature) into a zero-day exploit vector. Cloud hardware infrastructure is actively exploitable; software isolation (like virtualization) does not provide absolute defense if physical memory deduplication exposes underlying hardware defects.

### The FFS Attack Framework

Flip Feng Shui (FFS) converts hardware bit-flip vulnerabilities into a powerful, software-like arbitrary memory corruption primitive through three key stages: memory templating, memory manipulation via deduplication, and exploitation. **Memory Templating** fingerprints physical memory to identify reliable hardware bit-flip locations (templates), including offset and direction (1-to-0 or 0-to-1). **Physical Memory Spraying**, similar to heap spraying in software exploits, this technique precisely aligns target process data into attacker-identified vulnerable physical memory pages (templates). The OS uses a disk cache mapped into physical memory to speed up file access; data stored in the page cache is directly susceptible to FFS attacks during read operations.

The theoretical framework of FFS transforms raw hardware bit flips into controllable attack vectors. The attack consists of three phases. First, **Memory Templating** uses double-sided Rowhammer algorithms to analyze and discover vulnerable pages and specific bit-flip offsets under attacker control. Second, **Memory Manipulation** exploits Kernel Same-page Merging (KSM). By constructing a malicious page identical to the target victim page, KSM naturally merges the victim's physical mapping with the attacker's pre-built, vulnerable physical page. Finally, **Exploitation** occurs by triggering a bit flip on this shared physical page. Since the selected target data resides in the victim's page cache (e.g., OpenSSH `authorized_keys` or apt `sources.list` and `trusted.gpg`), data corruption is equivalent to opening a backdoor, allowing unauthorized SSH access or malicious software updates—essentially providing full unprivileged control without relying on any software logic flaws.

The feasibility of this attack lies in converting hardware predictability (Rowhammer vulnerability + OS memory deduplication) into targeted software payloads without causing system crashes or general data corruption errors. The chain of dependencies requires KSM to merge attacker-provided pages into attacker-controlled physical locations (stable or unstable trees), making the OS an active participant in placing victim data at the point of attack.

### RSA Cryptanalysis through Bit Flips

A single bit flip in the RSA public key modulus $n$ transforms it into a new, predictable composite $n'$, which is highly likely to contain smaller prime factors. This allows an attacker to efficiently factor the new modulus and reconstruct a corresponding valid private key. **RSA Modulus ( $n$ )**: RSA's core security relies on $n = p_1 \cdot p_2$, where $p_1$ and $p_2$ are two large primes of roughly equal size. Factoring large standard $n$ values (e.g., 2048 bits) is computationally infeasible. **Factoring Complexity**: theoretical algorithms like the General Number Field Sieve (GNFS) or Lenstra's Elliptic Curve Method (ECM) are used to attempt integer factorization. Their runtime determines the security of the key length. **Carmichael Function ( $\lambda$ )**: used to calculate the private exponent $d$ from the public exponent $e$, which requires the prime factors of $n$.

RSA security relies entirely on the difficulty of finding the prime factors of the public key modulus $n$. Typically, $n$ is the product of two equal-sized large primes. If an attacker triggers a single bit flip in $n$, the resulting value $n'$ is corrupted. Since $n'$ is just an arbitrary odd number (or even, and even numbers are easy to factor), its prime factors follow standard statistical distributions rather than the forced "two large primes" structure.

Specifically, $n'$ is very likely to consist of several smaller prime factors. Because factoring algorithms like Lenstra's Elliptic Curve Method (ECM) calculate based on the size of the smallest prime factor rather than the size of $n'$ itself, factoring the tampered $n'$ becomes computationally feasible (taking hours or days instead of centuries). Once $n'$ is factored, the attacker can easily compute a forged private key $d'$ that mathematically matches the tampered public key stored in the victim's memory, completely breaking standard authentication checks and paving the way for forged signatures or decryption.

Integer form: every integer greater than 1 can be represented as a unique product of primes. This defines the general target state of the corrupted modulus $n'$. Unlike the original RSA $n$ (where $s=2$ and both primes are large), $n'$ may have a larger $s$ and smaller prime factors, changing the dynamic from "unbreakable" to "factorable."

$$
n' = \prod_{i=1}^{s} p_i^{\gamma_i}
$$

- $n'$: bit-flipped (corrupted) RSA modulus.
- $p_i$: distinct prime factors of $n'$, $p_1 > p_2 > \dots > p_s$.
- $\gamma_i$: multiplicity or power of prime $p_i$; for original RSA $n$, $\gamma_1 = \gamma_2 = 1$.
- $s$: number of distinct prime factors; for original RSA $n$, $s = 2$.

The Carmichael function of $n'$ defines how to compute its value given the prime factorization of a composite number. This is a key step in forging a new private key. An attacker can only calculate the forged private exponent $d = e^{-1} \pmod{\lambda(n')}$ after successfully determining the prime factors $p_i$ used to compute $\lambda(n')$.

$$
\lambda(n') = \text{lcm}( p_1^{\gamma_1 - 1} \cdot (p_1 - 1), \dots , p_s^{\gamma_s - 1} \cdot (p_s - 1) )
$$

- $\lambda(n')$: Carmichael function of $n'$, used for calculating the private exponent.
- $\text{lcm}$: Least Common Multiple.
- $p_i, \gamma_i$: as defined above.

General Number Field Sieve (GNFS) complexity: represents the asymptotic runtime for factoring $n$ using the best-known general algorithm (GNFS). The key parameter is the "1/3" in the exponent. It explains the security of the standard RSA algorithm, as GNFS time depends on an immense amount of operations. When the key length is $n$, cracking a 2048-bit key is too slow. $L_n$ is the L-notation complexity function, standard for sub-exponential time; $n$ is the integer to be factored.

$$
O(L_n[1/3, 1.92])
$$

where:

$$
L_n[a,b] = \exp( (b + o(1)) (\ln n)^a (\ln \ln n)^{1-a} )
$$

Lenstra's Elliptic Curve Method (ECM) complexity $L_{p_s}$: $p_s$ is the smallest prime factor sought. ECM's asymptotic runtime depends on the size of the _smallest prime factor_ ( $p_s$ ) it is currently trying to extract, not the size of the full integer $n$. This explains why the exploit works. Since the bit-flipped modulus $n'$ is essentially a random odd number, statistically its prime factors are small. Even if the total length of $n'$ is 2048 bits, ECM can quickly peel off smaller 40-100 bit prime factors, recursively reducing the count until full factorization.

$$
O(L_{p_s}[1/2, \sqrt{2}])
$$

**Probabilistic Dependency**: this mathematical attack relies on the Erdős-Kac theorem, which states that random integers have a statistical probability of having small prime factors. While a single flip is not guaranteed to succeed, for 1024-2048 bit moduli, flipped keys have a 12-22% chance of being easily factorable, making parallel attacks highly feasible. **Algorithm Choice**: the difference between GNFS (limited by $n$) and ECM (limited by $p_s$) is the core mathematical observation that makes the software exploit possible.

### Exploiting Linux KSM and THP

The practical implementation of dFFS exploits Linux Kernel Same-page Merging (KSM) for data deduplication and Transparent Huge Pages (THP) to efficiently perform double-sided Rowhammer attacks in KVM environments. This establishes an end-to-end attack against cryptographic keys like OpenSSH `authorized_keys` and GPG `trusted.gpg`. **Kernel Same-page Merging (KSM)** scans memory for identical pages and merges them into a single read-only physical page to save RAM. It uses "stable" (deduplicated) and "unstable" (candidate but not yet matched) red-black trees. **Transparent Huge Pages (THP)** automatically packs contiguous 4 KB pages into larger 2 MB "huge pages" to reduce TLB misses and improve performance. **Double-sided Rowhammer** is a more reliable and faster Rowhammer variant where an attacker rapidly reads two rows adjacent to the victim row (e.g., reading row $i-1$ and $i+1$ to attack row $i$).

How does **KSM** perform memory manipulation? If an attacker constructs a page matching a victim's page, KSM merges them. Attackers can exploit KSM's merging mechanism (relying on unstable tree timing or stable tree logic) to control the exact physical location of this shared page.

Second, performing Rowhammer attacks in KVM: since the guest OS is unaware of the host's physical memory layout, the attacker exploits THP (page mapping). THP maps guest virtual memory to host physical memory by allocating large contiguous 2 MB blocks. Since huge pages are always aligned to 2 MB boundaries, attackers can know the relative alignment of DRAM rows within that page, allowing precise targeting for double-sided Rowhammer without restricted access to `/proc/self/pagemap`.

There is a subtle interaction here: if KSM deduplicates a page within a 2 MB contiguous block, it splits the THP into 4 KB pages, breaking the Rowhammer setup. Attackers avoid this by writing slightly different unique entropy in the initial phase, hidden from KSM until the memory templating phase is complete.

Finally, the end-to-end attack:

- **OpenSSH Attack**: the attacker triggers the victim VM to cache its `authorized_keys` file, deduplicates it, flips a bit in the RSA modulus, factors the new weak modulus using the methods in Section 3, generates a matching private key, and logs in.
- **GPG/Apt Attack**: the attacker flips a bit in the cached `sources.list` file to redirect package downloads to a malicious URL (`ubunvu.com`), then flips a bit in the `trusted.gpg` file, factors the corrupted key, generates a matching private key, and signs malicious update packages. A flaw in GPG causes it to ignore self-signature checks when a bit-flipped key changes its Key ID, allowing for forged keys.
- **THP and KSM Interaction**: the observation that KSM prioritizes memory reduction over TLB efficiency (thus severely breaking THP) is a brilliant exploit insight. It reveals how two different memory management optimizations conflict and how an attacker must carefully control entropy to prevent KSM from interfering with the THP structure required for Rowhammer.
- **Side Effects of Corruption**: the discovery that GPG's internal mechanism for verifying key integrity (self-signatures) silently fails and skips verification when a Key ID (hash of the modulus) changes due to a bit flip is a profound illustration of software failing precisely because "hardware bit flips are not in its threat model."

### Defense Strategies and Mitigations

Mitigating Flip Feng Shui requires a defense-in-depth strategy, including hardware fixes (e.g., Targeted Row Refresh (TRR)), OS-level changes (e.g., disabling or randomizing memory deduplication), and software-level practices (e.g., implementing strong integrity checks for data read from the page cache). **Targeted Row Refresh (TRR)**, a hardware mitigation in newer RAM (like LPDDR4), counts row activations and proactively refreshes adjacent rows before leakage causes bit flips. **Zero-page Deduplication** is a restricted form of KSM where the OS only deduplicates memory pages entirely filled with zeros (common, but harder for attackers to exploit for tampering specific data).

Three levels of defense: Hardware, OS, and Software.

At the **hardware level**, while ECC memory prevents single-bit flips, Rowhammer can sometimes induce multiple bit flips in the same word, bypassing standard ECC. The true hardware solution is Targeted Row Refresh (TRR) in newer DDR4 or LPDDR4 memory, which counts activations and proactively refreshes vulnerable rows. However, replacing all existing DDR3 infrastructure is economically infeasible.

At the **OS layer**, the simplest fix is to completely disable KSM memory deduplication, breaking the memory manipulation primitive. However, this causes massive memory loss for cloud providers (up to 79% savings). Restricting KSM to "zero-page" deduplication retains about 46% savings but is still costly. A "simulated" randomized deduplication allocator attempts to break predictability, but sophisticated attackers can still force predictable allocations by triggering near out-of-memory (OOM) states.

At the **software layer**, a paradigm shift is needed: applications must assume that data read from the filesystem (via the page cache) is susceptible to bit-level tampering by an attacker. Software should implement strong integrity hashing immediately before using sensitive data (like public keys) or format files with self-verifying cryptographic signatures. Essentially, software threat models must now include "attacker-controlled bit flips."

**Security Economics**: the conflict between security and cost is perfectly illustrated here. The most effective software mitigation (disabling KSM) causes significant memory waste for cloud providers, so it cannot be universally disabled easily. **Threat Model Evolution**: presents a compelling philosophical argument to the software community: trust boundaries cannot stop at the filesystem. If data passes through physical memory, it must be treated as potentially tampered with by a co-hosted attacker.

## 3. CLKSCREW: Software-Based Fault Injection via DVFS

CLKSCREW is a novel pure-software fault attack that exploits security-oblivious power management mechanisms (Dynamic Voltage and Frequency Scaling, DVFS) to induce erroneous computations, bypassing hardware-enforced isolation (like ARM Trustzone) without physical access to the device. **Compared to Rowhammer**, while CLKSCREW shares the same essence as Rowhammer (software-induced hardware transient faults), they are fundamentally different because CLKSCREW targets microarchitectural operations at the power management layer, potentially making mitigations more complex. Modern power management and hardware security have many overlapping areas. **Dynamic Voltage and Frequency Scaling (DVFS)**, a power management technique, dynamically adjusts processor supply voltage and clock frequency to balance performance and energy consumption ( $P \propto V^2 f$ ). **Fault Injection Attack**, compromises security by inducing errors (e.g., via voltage glitches or clock manipulation) during processor execution, leading to predictable but incorrect behavior. **ARM Trustzone** provides hardware-isolated "Secure World" for sensitive computations, isolating it from the "Normal World" (e.g., Android). **Security-Oblivious Design** focuses on function or performance optimization (like energy efficiency), while ignoring security implications of exposing hardware controls to software.

The pursuit of energy efficiency has led to aggressive, hardware-software co-designed energy management mechanisms, particularly DVFS. These mechanisms allow software fine-grained control over critical hardware parameters (voltage and frequency). However, these interfaces are "security-oblivious," meaning they are designed for efficiency without considering potential security vulnerabilities.

The **CLKSCREW** fault attack exploits these software-exposed controls to push the processor beyond its reliable operating limits (overclocking or undervolting), inducing faults. Unlike traditional fault attacks requiring physical access, specialized equipment (lasers, thermal), or hardware modification, CLKSCREW is conducted entirely via software (e.g., a malicious kernel driver). This was demonstrated by extracting AES keys from ARM Trustzone on a standard Nexus 6 device and loading self-signed code.

CLKSCREW makes remote exploitation of fault attacks possible, a stark departure from traditional fault attacks requiring physical access. It is a design-level vulnerability; the root cause is not a simple bug but a fundamental design choice in energy management systems. Trustzone isolation: this attack specifically targets the assumption that software in the "Normal World" cannot affect the integrity or confidentiality of the "Secure World."

Mechanisms exploited by CLKSCREW:

- **Dynamic Power and Energy**: Power is the rate of energy consumption ( $P \propto V^2 f$ ), while energy is the total consumption over time ( $E = \int P dt$ ).
- **Power Management Integrated Circuit (PMIC)**: a dedicated hardware module (e.g., PMA8084) that regulates voltage for various SoC components.
- **Subsystem Power Manager (SPM)**: a hardware module providing a memory-mapped interface for software to communicate with the PMIC.
- **Phase-Locked Loop (PLL)**: a circuit that generates clock signals by multiplying a reference frequency (e.g., 19.2MHz) to a target frequency (e.g., 2.65GHz).
- **CPUfreq Governor**: an OS-level service (in Linux) that monitors system load and requests voltage/frequency changes via kernel drivers.

### Energy Management and DVFS Principles

DVFS balances energy consumption and performance through a hardware-software co-designed stack, where software-controlled regulators (voltage via PMIC, frequency via PLL) manage core operating parameters. Energy efficiency depends on the trade-off between speed and power. The hardware/software stack required for DVFS:

- **Hardware Regulators**: Voltage is managed by PMIC via SPM, and frequency is managed by PLL (specifically HFPLL on Nexus 6).
- **Drivers**: Vendor-specific drivers expose control to the OS.
- **OS Services**: CPUfreq governors initiate changes based on real-time feedback (e.g., system utilization).

Crucially, for efficiency and low latency, software is granted **full access** to these regulators. This sets the stage for the CLKSCREW attack.

Total energy consumed by the system is the accumulation (integral) of power consumed at each moment over a specific period. Reducing instantaneous power ( $P_t$ ) or required time ( $T$ ) reduces total battery energy consumption. This explains the fundamental motivation of energy management systems to minimize $P_t$ via voltage and frequency scaling.

$$
E_T = \int_{0}^{T} P_t dt
$$

- $E_T$: total energy consumption over time $T$
- $P_t$: instantaneous dynamic power at time $t$
- $T$: total duration

Dynamic Power: in digital circuits with a fixed capacitive load, power consumption is proportional to the square of the supply voltage and linear with frequency. Since power is proportional to the square of voltage ( $V^2$ ), reducing voltage is more effective for power savings than just reducing frequency. This explains why DVFS systems prioritize "undervolting" to save energy, which is a key primitive used in CLKSCREW.

$$
P_t \propto V_t^2 \times F_t
$$

- $P_t$: instantaneous dynamic power at time $t$
- $V_t$: supply voltage at time $t$
- $F_t$: operating frequency at time $t$

Clock Frequency: the core clock frequency is determined by a fixed reference clock multiplied by a variable $N$ configured in the HFPLL. This shows that software can change core speed by writing a new integer $N$ to a memory-mapped register. CLKSCREW exploits this to configure $N$ far beyond "safe" ranges (overclocking).

$$
F_{out} = N \times 19.2 \text{ MHz}
$$

- $F_{out}$: output core frequency
- $N$: multiplication ratio, adjustable via PLL registers
- $19.2 \text{ MHz}$: reference base frequency, crystal oscillator

**Software Exposure** emphasizes that frequency and voltage are controlled via **memory-mapped registers** accessible to privileged software. **Asynchronous Management** notes that DVFS cannot be fully implemented in hardware because it requires high-level OS context (like task scheduling), explaining the necessity of software-visible interfaces. **Nexus 6 Architecture**, independent clock domains per core, are considered a key feature, allowing an attacker core to manipulate a target core's frequency independently.

### Inducing Faults by Violating Timing Constraints

CLKSCREW faults are induced by software-controlled overclocking or undervolting operations that violate circuit timing constraints, exploiting the absence of hardware protection mechanisms and the isolation provided by multi-core voltage/frequency domains. Basic physical principles of digital circuit faults:

- **Synchronous Digital Circuits**: circuit operation is coordinated by a global clock signal.
- **Flip-Flops (FF)**: storage elements used to hold state; they capture input data on the rising edge of the clock signal.
- **Propagation Delay ( $T_{max\_path}$ )**: the time required for a signal to pass through combinational logic between two flip-flops.
- **Timing Constraint**: the clock period must be long enough to ensure data stabilizes at the destination before the next clock pulse.
- **Timing Violation**: data fails to "latch" correctly due to an over-fast clock (overclocking) or slow circuit (undervolting).

A gap exists between high-level software control and low-level hardware physics; digital logic relies on signal stability within a clock cycle. An attacker can shorten the clock period ( $T_{clk}$ ) or increase propagation delay ( $T_{max\_path}$ ) via undervolting, causing "bit flips" where the target flip-flop captures stale data.

The attack methodology includes four "general attack enablers":

- **No Hardware Protections**: regulators allow configurations beyond vendor-specified safety ranges.
- **Frequency/Voltage Interdependence**: lowering voltage makes circuits more sensitive to overclocking.
- **Cross-Core Isolation**: attacks can be confined to a single victim core while the attacker core remains stable.
- **Security-Oblivious Domains**: regulators operate across Trustzone boundaries.

Timing Constraint: the total time for a signal to leave one storage cell, pass through logic, and reach the next must be less than or equal to the interval between clock pulses. If $T_{clk}$ decreases (overclocking) such that the inequality fails, the second flip-flop will capture the _previous_ state instead of the _new_ one, leading to computational error. Timing constraints are the target of CLKSCREW attacks. The attack aims to break this inequality by reducing the left side ( $T_{clk}$ ) or increasing the right side ( $T_{max\_path}$ ) via undervolting.

$$
T_{clk} \ge T_{FF} + T_{max\_path} + T_{setup} + K
$$

- $T_{clk}$: clock period, $1 / \text{frequency}$
- $T_{FF}$: internal flip-flop delay, time for a signal to leave the source flip-flop
- $T_{max\_path}$: maximum propagation delay, logic delay between flip-flops
- $T_{setup}$: setup time, stability time before the clock signal
- $K$: microarchitectural constant, system-specific margin

Attack Parameters: A CLKSCREW attack is triggered by a specific sequence of voltage and frequency operations while the victim executes a program. It provides a formal way to find the "sweet spot" where faults are transient enough to corrupt data but stable enough not to crash the whole system. This set of parameters defines the search space for experimental results in subsequent sections (AES and RSA attacks).

$$
F_{\theta|T_{anchor}} = \{F_{volt}, F_{pdelay}, F_{freq\_hi}, F_{dur}, F_{freq\_lo}\}
$$

- $F_{\theta}$: fault parameter set characterizing the attack
- $T_{anchor}$: timing anchor, the execution point to trigger the fault
- $F_{volt}$: baseline operating voltage
- $F_{pdelay}$: pre-fault delay, fine-tuning loop
- $F_{freq\_hi}$: target high frequency, overclocked frequency
- $F_{dur}$: fault duration, number of high-frequency cycles
- $F_{freq\_lo}$: recovery frequency, baseline frequency

Injecting faults into a victim without affecting the attacker's own core is difficult. This is solved by pinning threads to different cores (Attack Enabler #3). **Continuous Instability Curve** shows there is no single "crash point" but a curve of potential fault-inducing configurations, giving attackers significant freedom. For **Hardware "Agnosticism"**, regulators (PMIC/PLL) do not distinguish between "Secure World" and "Normal World" code, which is the root cause of why Trustzone cannot defend against such attacks through isolation alone.

### Case 1: AES Key Extraction from Trustzone

Secret AES keys stored in Trustzone are extracted by inducing single-byte faults in specific AES rounds via software-controlled DVFS, then applying Differential Fault Analysis (DFA) to the resulting erroneous ciphertexts. **Differential Fault Analysis (DFA)**, a side-channel attack, compares correct outputs of a cryptographic operation with erroneous ones (fault-induced) to recover keys. **AES-128** Advanced Encryption Standard using a 128-bit key, consists of 10 rounds of transformations (SubBytes, ShiftRows, MixColumns, AddRoundKey). **Hardware Cycle Counter (CCNT)**, an ARM architecture register, records clock cycles since the last reset, here used to profile the duration of Trustzone calls. **Secure Monitor Call (SMC)** is used to switch the processor from Normal World (non-secure) to Secure World (Trustzone).

This attack targets an AES decryption app running inside Trustzone, even though the "Normal World" cannot read the AES key. The attacker can directly use the key to trigger decryption of known ciphertexts and observe the decrypted plaintexts. The attack uses **CCNT** to measure the execution time of Trustzone decryption. By observing timing stability (~840k cycles), the attacker can precisely target the fault during the 7th or 8th round of AES. By finding the "sweet spot" of frequency and duration (680 cycles at 3.69 GHz), the attack can introduce a single-byte random corruption. This specific fault model allows using existing DFA algorithms to reduce the key search space from $2^{128}$ to $2^{12}$, enabling easy brute-force.

Parameter set $F_{\theta, AES-128}$: the "magic numbers" found experimentally on the Nexus 6 to reliably induce a single-byte fault in the 8th AES round. This shows that theoretical "timing violations" can be controlled to target specific code windows (one AES round) and specific error granularities (one byte). This proves the feasibility of the timing anchor strategy even without advanced cache-based profiling for this specific attack.

$$
F_{\theta, AES-128} = \{F_{volt} = 1.055\text{ V}, F_{pdelay} = 200\text{k}, F_{freq\_hi} = 3.69\text{ GHz}, F_{dur} = 680, F_{freq\_lo} = 2.61\text{ GHz}\}
$$

- $F_{volt}$: baseline voltage, 1.055V
- $F_{pdelay}$: pre-fault delay loop count, 200,000 iterations
- $F_{freq\_hi}$: overclock frequency, 3.69 GHz
- $F_{dur}$: fault duration, 680 iterations
- $F_{freq\_lo}$: recovery frequency, 2.61 GHz

**Trustzone Timing Leakage (Enabler #5)**: being able to read CCNT across secure boundaries is a critical vulnerability, as it allows attackers to build timing profiles of secure operations. By varying the pre-fault delay ( $F_{pdelay}$ ), attackers can "precisely" control which specific AES round is corrupted. An average of only 20 attempts is needed to obtain a valid faulty ciphertext, indicating CLKSCREW is a practical and efficient attack vector.

### Case 2: RSA Signature Bypass for Malicious Firmware

CLKSCREW breaches RSA-based firmware signature verification in Trustzone by introducing errors during the endianness flipping of the public key modulus $N$, generating a factorable "faulty" modulus $N_A$, which attackers exploit to generate valid signatures for arbitrary code. This complex integrity attack involves:

- **RSA Signature Chain**: where each firmware image is signed with a private key and verified using a public key modulus ( $N$ ) embedded in a certificate chain.
- **Montgomery Multiplication**: an efficient modular multiplication algorithm commonly used in RSA implementations to avoid costly division steps.
- **Prime+Probe Cache Attack**: a side-channel attack where an attacker "fills" cache sets with their own data and "probes" if the victim's execution has evicted any, enabling high-precision timing of victim code.
- **Pollard's $\rho$ and ECM Factorization**: algorithms used to factor large composites into prime factors. If modulus $N$ is tampered into $N_A$ with smaller prime factors, the RSA system fails under that tampered state.

The target is the Trustzone bootloader's firmware verification process. The core of the attack lies in injecting a fault during a function named `FLIPENDIANNESS`, which processes the public key modulus $N$ before RSA decryption. The attack has two main phases:

- **Fault Discovery**: attackers use cross-core **Prime+Probe** cache analysis to identify the exact moment the bootloader verifies the last certificate in the chain. A fault is then induced, corrupting $N$ into a new value $N_A$.
- **Cryptanalysis**: if $N_A$ is factorable (likely if the fault results in smaller prime factors), the attacker derives a new private key $d_A$ corresponding to $N_A$. A special signature $S'_A$ is then constructed such that, when processed by faulty hardware, it generates the hash of their malicious code.

RSA Verification: code $C$ is considered authentic only if its hash matches the result of decrypting the signature with the public key. This is the gatekeeper for code allowed to run in Trustzone. CLKSCREW bypasses this by changing $N$ to a value the attacker "owns."

$$
Sig^e\mod N == H(C)
$$

- $Sig$: code signature included in the update package
- $e$: public key exponent, typically 0x10001
- $N$: public key modulus, the trust anchor
- $H(C)$: SHA-256 hash of code $C$, calculated at runtime

Montgomery multiplication performs multiplication followed by division by $N$ but uses a base $r$ for efficiency in digital hardware. The attack signature $S'_A$ must account for $N$ being used for Montgomery setup while the corrupted $N_A$ is used for other parts. By using these Montgomery terms, the expected output of a normal RSA operation is equated with the output of the faulty one to solve for $S'_A$.

$$
\text{MONTMULT}(x, y, N, r^{-1}) \leftarrow x \cdot y \cdot r^{-1} \mod N
$$

- $x, y$: input operands
- $N$: modulus
- $r$: Montgomery radix, typically $2^{2048}$ for 2048-bit RSA
- $r^{-1}$: modular inverse, $r \cdot r^{-1} \equiv 1 \mod N$

Attack Signature Derivation: given a specific corruption $N \to N_A$, this linear congruence relation allows attackers to find the precise bits needed in the "attack signature" to yield the desired malicious hash. It proves the attack is not a simple "crash" but a precise tampering of cryptographic logic.

$$
S'_A \cdot K \mod N_A \equiv x_{desired} \mod N_A
$$

- $S'_A$: attack signature, the target value to solve for
- $K$: composite constant, $(r^2 \mod N) \cdot r_A^{-1}$
- $N_A$: faulty modulus, the corrupted value of $N$
- $x_{desired}$: target state leading to $H(C_{attack})$

**Algorithm `DECRYPTSIG(S, e, N)`** performs RSA modular exponentiation using Montgomery multiplication. **Fault injected at** line 4 causes `Nrev` (and all subsequent modular operations) to be corrupted. The variable `R` (line 15) is calculated using the _original_ $N$ from line 3. Therefore, attackers cannot simply use a standard RSA signature for $N_A$; they must use the derivation in Eq. 5.3 to compensate for the "mixed" state of uncorrupted `R` and corrupted `Nrev`.

### Cross-Stack Mitigation Challenges

Preventing CLKSCREW attacks requires complex cross-stack mitigations, as simple hardware limits or software redundancies are either insufficient or negate the energy efficiency benefits DVFS aims to provide. Potential defenses against this attack:

- **P-states and C-states**: CPU performance states (P-states, changing frequency/voltage) and operating states (C-states, changing power modes like sleep), used in both mobile and cloud environments (e.g., AWS EC2).
- **eFuses**: hardware fuses that can be blown during manufacturing to permanently set operating limits (e.g., max voltage).
- **Process Variation**: naturally occurring physical differences during semiconductor manufacturing that cause two identical chips to have slightly different safe operating limits.
- **Rowhammer**: a well-known hardware vulnerability where rapid, repeated memory access causes bit flips in adjacent rows, often compared to CLKSCREW due to its software-to-hardware fault nature.

Broader applicability of CLKSCREW: similar energy management designs exist in modern Intel processors, ARMv8 architecture, and even cloud environments. Potential defenses:

- **Hardware Limits**: enforcing hard limits via eFuses is difficult because limits vary across chips due to process variation, and regulators are often designed to be portable across SoC models.
- **Cross-Boundary Regulators**: using separate physical regulators for Trustzone is costly and requires duplicating complex DVFS software stacks inside the Secure World.
- **Redundancy**: adding hardware logic redundancy or software execution redundancy incurs significant area and power overhead, defeating the purpose of mobile energy management.

A comprehensive redesign of energy management systems is needed, integrating security from the ground up at the architectural level, as the root cause is a fundamental, security-oblivious design decision rather than a simple bug. No "single, simple fix" or "piecemeal patch" exists because the vulnerability stems from deliberate, practical engineering decisions aimed at maximizing efficiency. AWS exposes P-states and C-states to EC2 instances, suggesting CLKSCREW-style attacks could apply to multi-tenant cloud environments, posing a cross-VM attack risk.

## 4. Hertzbleed: Remote Timing Attacks via Frequency Scaling

Modern frequency-scaling processors invalidate the core assumption of constant-time programming, as data-dependent power consumption can be remotely observed through changes in CPU frequency. **Constant-time Programming (Fallacy)** ensures an algorithm uses a fixed number of CPU cycles for every execution is sufficient to prevent timing leaks. Hertzbleed proves this assumption wrong when the clock speed (frequency) itself varies. **Remote Timing Analysis** infers secret data by measuring the execution time of a target service over a network.

Hertzbleed shows that data-dependent power variations in modern x86 CPUs can be converted into remote timing attacks via Dynamic Voltage and Frequency Scaling (DVFS), breaking constant-time cryptographic implementations. The **Hertzbleed** attack combines **side-channel analysis** and **CPU power management**.

- **Power Side-Channel Attack**: attacks that exploit power consumption variations to infer data processed by the CPU.
- **Dynamic Voltage and Frequency Scaling (DVFS)**: a power-saving feature that adjusts processor clock speed (frequency) and voltage based on workload and thermal limits.
- **Timing Attack**: a side-channel attack where the execution time of an algorithm depends on secret data.
- **Constant-time Programming**: a defensive technique where code is written so its execution time is independent of the data processed (e.g., no data-dependent branches or memory lookups).

Modern CPU frequency scaling (DVFS) depends on data because it responds to power consumption, which itself is data-dependent. This means even "constant-time" code—whose execution cycles are independent of data—can have variable _actual execution time_ because the clock frequency itself changes:

- Reverse engineering the relationship between data bit patterns (e.g., Hamming weight and bit positions) and CPU frequency.
- Demonstrating a practical remote chosen-ciphertext attack against **SIKE** (Supersingular Isogeny Key Encapsulation), a post-quantum cryptographic algorithm, by amplifying subtle power differences into measurable timing signals.

**Granularity Challenge**: frequency channels are coarse-grained (millisecond scale), typically making them harder to exploit than microarchitectural timing channels (e.g., cache attacks). The attack applies to modern Intel and AMD x86 CPUs. It "lifts" power attacks into the timing domain, removing the need for physical probes or software power measurement interfaces (like RAPL).

### The Hertzbleed Mechanism

Hertzbleed is a novel side-channel attack that exploits frequency scaling induced by DVFS to leak secret data through remote execution timing, bypassing traditional power analysis mitigations. **Hamming Weight (HW)** is the number of bits set to "1" in a binary string (e.g., $HW(1011_2) = 3$). **Hamming Distance (HD)** is the number of positions where corresponding bits differ between two binary strings (e.g., $HD(101, 110) = 2$, as bits 0 and 1 differ). **Post-Quantum Cryptography (PQC)** is designed to resist quantum computer attacks; SIKE is a specific type of PQC. **Isogeny-based Cryptography**, a branch of PQC, exploits mappings (isogenies) between elliptic curves.

While software-based power measurement interfaces (like Intel RAPL) have been restricted to mitigate side-channel leakage, Hertzbleed makes these restrictions moot. **DVFS** adjusts clock frequency based on power/thermal limits, bridging data-dependent power and actual execution time.

Frequency vs. Time: 1 Hz = 1 cycle/second. Clock frequency determines the cycles occurring per second of wall-clock time. If an algorithm always takes $N$ cycles (cycle-constant) but the frequency $f$ drops due to high power consumption, the actual runtime $T = N/f$ increases. This is the fundamental "transitivity" of converting power leakage into timing loss. Since DVFS response is at the millisecond scale, "amplification" (repeated execution of the same data-dependent operation) is required to make the signal detectable. Hardware effects are "non-uniform," meaning the position of "1"s in a 64-bit word affects power consumption, an important finding for side-channel modeling.

### CPU Power Management and Leakage Models

The technical foundations of CPU power management (P-states, Hardware-Controlled Performance States (HWP), Thermal Design Power (TDP)) and traditional power side-channel leakage models (Hamming weight and Hamming distance) for understanding data-dependent power.

- **Hardware-Controlled Performance States (HWP / Speed Shift)**: a feature where the CPU autonomously manages its P-states to optimize responsiveness and power rather than relying on the OS.
- **Thermal Design Power (TDP)**: the maximum heat a CPU is expected to dissipate under sustained load.
- **Turbo Boost**: a technology allowing CPUs to run at speeds higher than their base frequency for short bursts, provided they stay within temperature and power limits.
- **Leakage Models (HW and HD)**: mathematical approximations describing the relationship between data and power consumption.
- **Hamming Weight (HW)**: power consumption is proportional to the number of bits set to "1".
- **Hamming Distance (HD)**: power consumption is proportional to the number of bit toggles (0-to-1 or 1-to-0) between consecutive states.

Modern Intel CPUs manage frequency and power, transitioning from OS-managed frequency (**SpeedStep**) to CPU-autonomous management (**HWP**). Crucially, CPUs can exceed their **TDP** in "Turbo Boost" mode, but if a workload is sustained, the CPU eventually "throttles" (reduces frequency) to stay within TDP limits. This throttling mechanism is the physical trigger for the Hertzbleed timing signal. Finally, while power side-channel attacks were once limited to physical proximity or restricted software interfaces, Hertzbleed aims to remove these constraints.

Hamming Distance (HD): power consumption is proportional to the number of bits changed between two consecutive clock cycles. In CMOS circuits, charging and discharging capacitors (which occurs during bit flips) consumes the most energy. Hertzbleed exploits this, showing that high-activity data (many toggles) leads to faster thermal throttling.

$$
P(D_{prev}, D_{curr}) \approx k \cdot HD(D_{prev}, D_{curr}) + b
$$

- $P$: power consumption
- $D_{curr}$: current data value
- $D_{prev}$: previous data value
- $HD$: Hamming distance function, bit transition count
- $k, b$: constants, gain and offset

Hamming Weight (HW): power consumption is proportional to the total number of '1' bits in a data word. Even without toggles, keeping certain lines at high voltage causes leakage current or different resistive losses. The Hamming weight function's impact on frequency varies by bit position.

$$
P(D) \approx k \cdot HW(D) + b
$$

**P-state Naming Confusion**: Linux and Intel have opposite conventions for numbering P-states (Intel calls higher-frequency P-states "low" states, while Linux does the opposite). The Linux convention is followed here (higher P-state = higher frequency). Timing signals only stabilize when the CPU reaches a "steady state" (throttled to TDP), meaning the attack requires a warmup period.

### Profiling the CPU Frequency Leakage Channel

Instruction execution and CPU frequency leakage: the data values processed (specifically their Hamming weights) directly affect the CPU frequency distribution after the processor reaches a power-limited steady state.

- **MSR (Model Specific Register)**: hardware registers used to toggle or monitor specific CPU features. `MSR_IA32_MPERF` and `MSR_IA32_APERF` are used here to calculate "effective" frequency.
- **RAPL (Running Average Power Limit)**: Intel's interface for reporting and limiting energy consumption.
- **Steady State (Power Management)**: the equilibrium state where the CPU has exhausted its thermal/power "budget" for Turbo Boost and has throttled down to its TDP limit.
- **Effective Frequency**: calculated as the ratio of actual cycles (`APERF`) to reference cycles (`MPERF`) over a time interval.

Experimental setups on Intel CPUs (from Coffee Lake to Tiger Lake) prove that "heavy" workloads (e.g., floating-point) reach power limits faster and result in lower steady-state frequencies than "light" ones (e.g., integers). More importantly, it proves **data is the primary factor affecting frequency**: executing the same `shlx` (shift left) instruction in a loop but with different Hamming weights (16, 32, or 48 bits) results in different frequency distributions. High HW data consumes more power, causing the CPU to reach steady state faster and maintain a lower average frequency.

Effective Frequency Calculation: the actual frequency of a CPU is the base frequency multiplied by the ratio of cycles actually executed to the cycles that would have occurred at a fixed reference frequency. Since frequency fluctuates rapidly, sampling these registers over a time window (e.g., 5 ms) provides the average frequency within that window. This measurement technique allows observing subtle frequency shifts induced by data processing.

$$
f_{eff} = f_{base} \cdot \frac{\Delta \text{APERF}}{\Delta \text{MPERF}}
$$

- $f_{eff}$: effective frequency, measured clock speed
- $f_{base}$: base frequency, nominal non-turbo speed
- $\Delta \text{APERF}$: actual performance cycles, increment at current speed
- $\Delta \text{MPERF}$: max performance cycles, increment at fixed speed

**Sampling Interval** 5 ms confirms that while frequency side-channel attacks are relatively slow compared to cycle-accurate timing attacks, they are sufficient for practical clock measurements. For low-power instructions, a "fixed workload" must run in the background to push the CPU into a power-limited steady state, indicating the attack requires the CPU to be near its power/thermal ceiling.

### Modeling ALU Power Consumption

Reverse engineering power leakage on Intel ALUs proves that Hamming distance (bit transitions) and Hamming weight (bit counts) contribute independently, cumulatively, and non-uniformly (bit-position dependent) to CPU power and frequency.

- **ALU (Arithmetic Logic Unit)**: the foundational component of a CPU core responsible for basic math (add, sub, mul) and logic (AND, OR, XOR).
- **Execution Ports**: modern CPUs have multiple "ports" that can execute instructions in parallel. Instructions are interleaved (e.g., using ports 0 and 6 for shift operations) to maximize throughput and isolate leakage.
- **Instruction-level Power Profiling**: a method of measuring the "cost" of data-dependent operations by looping specific instructions.
- **Non-uniform Hamming Weight**: the discovery that bits set to "1" in the Most Significant Bits (MSB) consume different power than those in the Least Significant Bits (LSB).

Two primary leakage effects on modern Intel CPUs:

- **Hamming Distance (HD)**: by using shift operations (`shlx`/`shrx`) to move bit patterns without changing the total count of "1"s, power is proven to grow linearly with the number of 0-to-1 and 1-to-0 transitions.
- **Hamming Weight (HW)**: using the `or` instruction with fixed inputs (no transitions) proves that more "1"s in a register consume more power. Crucially, this effect is **non-uniform**, with the most significant 32 bits of a 64-bit word having a different power impact than the lower 32 bits.

Finally, **Additivity** proves that both effects occur simultaneously and their impact on frequency can be summed, allowing for complex leakage modeling.

Total power is the sum of a constant baseline, a term proportional to bit transitions, and a weighted sum of bits set to "1". The total power leakage explains why different data patterns with the same "weight" still leak different amounts of information. It confirms these terms are additive ( $P_{total} = P_{HD} + P_{HW}$ ), which is critical for attackers to predict frequency drops.

$$
P_{total}(D_{prev}, D_{curr}) = P_{base} + \mathcal{F}_{HD}(HD(D_{prev}, D_{curr})) + \sum_{i=0}^{n-1} w_i \cdot b_i
$$

- $P_{total}$: total power driving frequency scaling
- $P_{base}$: baseline power, power of empty loop
- $\mathcal{F}_{HD}$: HD leakage function, transitions proportional to bit toggles
- $w_i$: weight of the $i$-th bit position, non-uniformity
- $b_i$: value (0 or 1) of the $i$-th bit, input data
- $n$: word length (64), register width

**HD Transmitter**: loops shift operations on a fixed pattern. By varying `rax` (shift count), an attacker can change the number of overlapping/differing bits between `shlx` and `shrx` results without changing the total "1"s in the system. It isolates the **Hamming distance** effect. Increasing `COUNT` increases toggles in ALU outputs, leading to higher power and lower frequency.

<details><summary>Code</summary>

```assembly
rax = COUNT
rbx = 0x0000FFFFFFFF0000
loop:
 shlx %rax,%rbx,%rcx // rcx = rbx << rax
 shlx %rax,%rbx,%rdx // rdx = rbx << rax
 shrx %rax,%rbx,%rsi // rsi = rbx >> rax
 ...
 jmp loop
```

</details>

**HW Transmitter** loops repeated OR operations on the same values. Since inputs and outputs are identical for every iteration, the bit transition count is zero ( $HD=0$ ). It isolates the **Hamming weight** effect. Any frequency change is necessarily due to the number of "1"s in the "LEFT" and "RIGHT" registers, not bit transitions. This led to the discovery of non-uniformity.

<details><summary>Code</summary>

```assembly
rax = LEFT
rcx = … = r11 = RIGHT
loop:
 or %rax,%rcx // rcx = rax | rcx
 or %rax,%rdx // rdx = rax | rdx
 ...
 jmp loop
```

</details>

- **Microarchitectural "Bow"**: hardware effects exhibit a "bow" shape, suggesting bit power is not only non-uniform but may vary non-linearly across a register.
- **Independence**: Finding 5 states that hardware effects for one operand are independent of the other, simplifying attack model construction.
- **AMD Comparison**: while focused on Intel, AMD CPUs have "similar leakage models," indicating a widespread architectural vulnerability.

### Remote Exploitation of SIKE

A practical remote timing attack against SIKE exploits "exceptional 0 values" in the Montgomery Three-Point Ladder algorithm to amplify 1-bit key guesses into measurable CPU frequency boosts, enabling full 378-bit key recovery.

- **SIKE (Supersingular Isogeny Key Encapsulation)**: a PQC protocol based on the difficulty of finding mappings (isogenies) between elliptic curves.
- **Montgomery Three-Point Ladder**: a robust constant-time algorithm used to compute point multiplication $P + [m]Q$ on an elliptic curve without leaking the scalar $m$ via traditional timing or power channels.
- **Projective X-coordinates**: representing points on a curve as $(X:Z)$, where the affine coordinate $x = X/Z$. This avoids costly modular inversions in intermediate steps.
- **Exceptional 0**: a specific failure state in the Montgomery ladder (specifically the `xADD` function) where invalid inputs (like a point of order 2) cause the output to become $(0:0)$, which is not a valid point.

**Chosen-Ciphertext Attack (CCA)**: by cleverly constructing points P and Q in the ciphertext and using previously recovered bits of the key $m$, if the guess for the $i$-th bit is correct ($m_i \neq m_{i-1}$), one can force the $(i+1)$-th round of the Montgomery ladder to produce an "exceptional 0" $(0:0)$. Once $(0:0)$ is produced, the ladder "gets stuck," and all subsequent operations occur on 0. This ladder and its subsequent isogeny evaluations are all based on 0. **Timing Signal** operations on 0 consume significantly less power than operations on large, random-looking field elements (SIKE-751 uses 188-byte numbers). This power reduction triggers a frequency _increase_, speeding up decapsulation. This was verified using **CIRCL** and **PQCrypto-SIDH** libraries, recovering full keys over a network in 36 and 89 hours, respectively.

Montgomery Invariant: at the start of the $i$-th iteration, internal registers $(X_0:Z_0), (X_1:Z_1), (X_2:Z_2)$ satisfy:

$$
(X_0 : Z_0) \sim [2^i]Q
$$

$$
(X_1 : Z_1) \sim P + [i]Q
$$

$$
(X_2 : Z_2) \sim (X_0 : Z_0) - (X_1 : Z_1)
$$

- $i$: integer value of the first $i$ bits of the key, $i = (m_{i-1}, \dots, m_0)_2$
- $P, Q$: points on an elliptic curve from the ciphertext
- $\sim$: projective equivalence, $(X:Z) \sim (rX:rZ)$
- $T$: point of order 2 (0,0), triggering `xADD` function failure

The ladder structure maintains three points where the third is always the difference between the first two. this structure allows constant-time bit processing. The attacker's target is $(X_2:Z_2)$. If it can be made equal to $T$, the `xADD` function inside the ladder will fail and output $(0:0)$.

Target Point Construction: to target bit $k$, if $m_{k-1} = 0$, the attacker sets $P \leftarrow [2^k - i]Q - T$, ensuring that at iteration $k$, the difference register $(X_2:Z_2)$ will exactly equal the "poison point" $T$. If the next bit $m_k = 1$, $T$ will trigger the "stuck at 0" state in a specific way. If $m_k = 0$, it won't, creating a binary timing difference.

**Three-Point Ladder (Simplified)**: iterates through every bit of the key. `xDBLADD` performs point doubling and addition simultaneously. If `X2:Z2` is $T$ (poison point) and $m_i=1$, the third parameter of `xDBLADD` (which is $T$) triggers the internal `xADD` fault. This implementation aims to be cycle-constant, but Hertzbleed's _frequency_ scaling exposes the internal data state.

**Fujisaki-Okamoto (FO) Check**: these malformed ciphertexts fail the FO consistency check. However, timing leakage occurs _before_ the check is completed, making FO protection ineffective against this specific side-channel. Due to network noise, attackers might misjudge a bit. Using a **backtracking strategy**, if no timing reduction occurs for 40 consecutive bits, it's assumed a previous bit was wrong. The primary fix is validating that input points $P, Q$ are correctly ordered and linearly independent before starting the ladder.

### Timerless Attacks and Covert Channels

Timerless attacks exploit the frequency side-channel to break KASLR or establish cross-process covert channels without high-precision timers by reading system-provided frequency metrics.

- **KASLR (Kernel Address Space Layout Randomization)**: randomly permutes the location of critical data regions in kernel address space to prevent buffer overflow attacks.
- **Prefetch Instruction (`prefetcht0`)**: CPU instructions indicating hardware to load a memory address into cache. They don't trigger page faults even on unmapped addresses, but their execution behavior varies based on address validity.
- **Covert Channel**: intentional communication paths exploiting side-channels (like frequency) to send messages between two processes that shouldn't communicate.
- **`cpufreq` Driver**: a Linux kernel component providing the `scaling_cur_freq` interface, allowing unprivileged users to see current CPU clock frequency.

Hertzbleed frequency phenomena are stable enough to be measured via standard OS interfaces, removing the need for hardware timers.

- **KASLR Break**: by executing `prefetcht0` on candidate kernel addresses, power (and frequency) varies based on whether the address is "mapped" (valid) or "unmapped" (invalid). Successfully recovered kernel base addresses on an i7-9700 in 2 minutes by scanning 512 possible offsets.
- **Covert Channel**: built a transmitter that modulates CPU frequency by switching between high-power (OR operations with high HW/HD) and low-power (SHLX operations with low HW/HD) loops. The receiver reads frequency to decode bits, achieving 30 bps with a 0.03% error rate.

Frequency Modulation (Transmitter Strategy): **Low Signal (Logic 0)**, high HW/HD OR instruction loop; **High Signal (Logic 1)**, low HW/HD `SHLX` instruction loop. The sender acts as a signal modulator, using CPU power management logic as a transmitter. This applies previous reverse engineering results: high power forces the CPU to throttle (signal low), while low power allows it to maintain or recover higher P-states (signal high).

**Privilege Level**: using the "unprivileged" `scaling_cur_freq` interface is a significant security risk, as it allows any user process to indirectly monitor the data intensity of other processes or the kernel. These attacks are **stealthy**, as they use legitimate architectural instructions (`prefetcht0`) and standard OS monitoring files that don't trigger security exceptions.

### Hardware-Level Defense Strategies

Hardware-level mitigations (like disabling Turbo Boost) are performance trade-offs that necessitate software-level blinding/masking techniques to remove the correlation between power and secret data.

- **Masking and Blinding**: defensive cryptographic techniques where secret data is combined with random values (blinding) or split into multiple shares (masking) so that any single intermediate calculation is statistically uncorrelated with the secret data.
- **Fully Homomorphic Encryption (FHE)**: a theoretical cryptographic "holy grail" allowing operations on encrypted data without decryption.
- **Disabling Turbo Boost / Hardware Power Modes**: a hardware-level mitigation forcing the CPU to run at a fixed lower frequency, eliminating frequency-scaling vulnerabilities.

Which CPUs are vulnerable? Intel Core CPUs are confirmed vulnerable. Older CPUs (Skylake/Kaby Lake) are harder to exploit as they only support turbo frequencies for single-core workloads, making it harder to reach power-limited steady states. **AMD Ryzen** CPUs are also vulnerable. On the defense side, disabling frequency scaling (Turbo Boost) is a "safe but slow" solution. Software-level mitigations are advocated:

- Validate inputs (as seen in SIKE fixes).
- Design "blinded cores" using HW/HD leakage models to interleave actual computations with "bit-inverted" versions to flatten power consumption.

**Performance vs. Security Trade-off**: disabling Turbo Boost significantly degrades system performance, so it's not ideal for many cloud providers. This is a **widespread vulnerability**, also present in AMD, indicating the problem is not a specific implementation flaw but an inherent trade-off in modern power management architecture.
