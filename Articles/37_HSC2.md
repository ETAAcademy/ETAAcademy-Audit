# From Stack Smashing to to Hardware-Enforced Security: The Long Search for Practical Memory Safety

Memory corruption has shaped systems security for decades. That design choice makes it possible for a single out-of-bounds access, use-after-free, or unintended disclosure to become an exploit primitive. The result is a long-running cycle in which attackers chain invalid reads and writes into code corruption, control-flow hijacking, data-only attacks, and information leaks, while defenders respond with increasingly specialized mitigation layers.

To follow that cycle from first principles to modern hardware support, it starts by modeling memory corruption as a staged attack pipeline and then grounds that model in the classic stack-based buffer overflow: stack-frame layout, saved frame pointers and return addresses, offset calculation, shellcode placement, NOP sleds, and environment-variable staging. It then examines why widely deployed defenses such as stack cookies, DEP/W⊕X, and ASLR raise the cost of exploitation without eliminating the underlying bug class, and why full pointer-precise memory safety remains difficult to deploy because of metadata overhead, binary compatibility, and the tension between protection, cost, and compatibility.

From there, the discussion shifts to hardware and policy mechanisms that try to recover stronger guarantees with lower deployment friction. CHERI redefines pointers as tagged architectural capabilities carrying bounds, permissions, and monotonic authority; MPK provides thread-local region switching through page keys and PKRU state; PAC attaches cryptographic authentication to pointers; MTE adds fine-grained lock-and-key allocation tags; CET hardens both backward and forward edges with shadow stacks and indirect-branch landing pads; and CFI formalizes the rule that execution must remain within a valid control-flow graph. The unifying theme is straightforward: the hardest problem is no longer imagining stronger defenses, but engineering protection mechanisms that real software stacks can actually afford to adopt.

## 1. Threat Model and Exploitation: Memory Corruption, Attack Pipelines, and Stack Overflows

## 1.1 Why Memory Corruption Remains the Core Systems Security Problem

Low-level software is still written in languages that expose raw memory operations. That means memory safety failures are not edge cases; they are a built-in risk of the programming model. The most useful way to understand the space is not as a list of bugs, but as a staged attack pipeline that turns one invalid access into a working exploit.

### The Anatomy of a Memory Corruption Attack

At its core, a memory corruption exploit is not a single bug but a multi-step pipeline, modeled formally as a staged progression towards the final attack payload:

$$
E = s_1 \rightarrow s_2 \rightarrow s_3 \rightarrow s_4 \rightarrow s_5 \rightarrow s_6
$$

The attack begins with creating an invalid pointer ($s_1$) - either a spatial error (out of bounds) or a temporal error (a dangling pointer to a freed object) - and dereferencing it ($s_2$). Once that invalid pointer is used, the exploit branches into specific attack vectors:

1. **Code Corruption:** Directly overwriting executable code bytes.
2. **Control-Flow Hijacking:** Corrupting code pointers (like return addresses or function pointers) to redirect execution flow.
   ```c
   func_ptr jump_table[3] = {fn_0, fn_1, fn_2};
   jump_table[user_input](); // If user_input is uncontrolled, this indexes out of bounds and hijacks control-flow
   ```
3. **Data-Only Attacks:** Modifying ordinary but security-critical state to alter program behavior without hijacking control flow.
   ```c
   bool isAdmin = false;
   // ... (memory corruption occurs nearby, flipping the boolean) ...
   if (isAdmin) // The attacker gains privileges, without executing a single hijacked instruction
   ```
4. **Information Leaks:** Reading unintended memory to bypass probabilistic defenses like ASLR by revealing secrets or memory layouts.
   ```c
   printf(user_input); // Format string vulnerabilities (e.g., passing "%3$x") allow arbitrary stack disclosure
   ```

An invalid read can be just as useful as an invalid write. Modern exploit chains often rely on disclosures first, then use the leaked state to defeat probabilistic protections later.

### Why Commonly Deployed Defenses Are Inherently Partial

Most everyday defenses can be formalized as a core set of deployed mitigations:

$$
P_{\text{deployed}} = \{\text{Stack Cookies},\ \text{SafeSEH/SEHOP},\ \text{DEP/W}\oplus\text{X},\ \text{ASLR}\}
$$

These mitigations matter, but they do not eliminate the underlying bug class. They only interfere with particular stages of exploitation.

- **Stack cookies** protect against contiguous stack buffer overflows that alter the return address. They rely on checking a canary value right before the function returns:

$$
cookie_{\text{return}} = cookie_{\text{entry}}
$$

They fail completely against direct indexed overwrites or attacks that target function pointers elsewhere in memory.

- **DEP (W⊕X)** stops the execution of injected "shellcode" but leaves attackers free to pivot to Return-Oriented Programming (ROP), reusing existing code snippets (gadgets) already present in executable memory.
- **ASLR** obscures where gadgets live. However, the effectiveness of ASLR relies entirely on secrecy:

$$
\text{ASLR effectiveness} \approx \begin{cases} \text{high}, & \text{if runtime addresses remain secret} \\ \text{low}, & \text{if information leakage reveals them} \end{cases}
$$

A single information leak collapses the entire protection scheme.

These defenses are not useless. They are simply narrow. They raise the cost of exploitation without removing the exploit primitive.

### Why True Memory Safety Is Hard to Ship

If those mitigations are incomplete, the obvious answer is full memory safety. In principle, memory safety means eliminating both spatial and temporal violations:

$$
\text{Memory Safety} = \text{Spatial Safety} \land \text{Temporal Safety}
$$

If both are prevented without false negatives, all memory-corruption exploits are stopped at the root.

- **Spatial Safety:** A dereference at address $a$ through pointer $p$ must remain within its exact legal bounds:

$$
base(p) \le a < bound(p)
$$

- **Temporal Safety:** For a pointer to be validly dereferenced, its uniquely referenced object ID must still be marked "live":

$$
valid(id(p)) = 1
$$

In practice, enforcing those guarantees precisely is expensive. Precise tracking requires pointer-based metadata. If developers attempt to optimize this by tracking metadata at the _object_ level instead of the _pointer_ level, they lose critical precision against intra-object corruption. For example, C permits traversing an entire structure footprint:

```c
memset(&strct, 0, sizeof(strct));
```

An object-based checker sees this as legal. But if a malicious pointer traverses outside an internal nested field while staying inside `strct`, the object bounds check will completely miss the violation.

To do it correctly, tools like SoftBound and CETS offer robust pointer-based coverage, but introduce immense overhead (often exceeding 110%) and face severe binary compatibility limits when hardened code interacts with unmodified legacy libraries.

### The Middle Ground: Integrity, Randomization, and Approximation

To escape the cost of full memory safety, researchers have explored intermediate designs:

- **Probabilistic Methods:** PointGuard XOR-encrypts pointers globally using a single key:

$$
p_{\text{enc}} = p \oplus K
$$

Data Space Randomization (DSR) randomizes variable representation entirely using individual masks:

$$
c_v = \mathrm{Enc}_{m_v}(v), \qquad v = \mathrm{Dec}_{m_v}(c_v)
$$

Both share a fatal flaw: they are only valid as long as attackers cannot guess or leak the secret mask.

- **Data and Data-Flow Integrity (DFI & WIT):** Instead of protecting every pointer, generic defenses focus on static analysis. Write Integrity Testing (WIT) restricts a pointer's write operations strictly to its statically computed points-to set:

$$
allow_write(p, o) \iff o \in Pts(p)
$$

Data-Flow Integrity (DFI) ensures a read is safe only if the last instruction to write the data belonged to the legal reaching-definition set:

$$
allow_read(r, \ell) \iff id(\ell) \in RD(r)
$$

While they protect control and non-control data better than DEP, static analysis is inherently approximate.

- **Control-Flow Hijack Defenses:** Control-Flow Integrity (CFI) monitors indirect control transfers, allowing jumps only to dynamically or statically approved targets. A shadow stack maintains perfect dynamic return integrity by duplicating saved returns:

$$
ret_{\text{program}} = ret_{\text{shadow}}
$$

In contrast, static CFI only enforces that the target belongs to an approximated legal set:

$$
allow_transfer(i, t) \iff t \in ValidTargets(i)
$$

Strong precise CFI demands accurate global tracking, which complicates dynamic linking.

### What Deployability Actually Requires

The harsh truth of memory defense technology is visible in what operating systems, compilers, and processors actually ship. The deciding factor is not theoretical elegance alone:

$$
\text{Practicality} = f(\text{Protection},\ \text{Cost},\ \text{Compatibility})
$$

Any defense that hopes to see broad deployment must balance three constraints:

1. **Robust Protection:** High accuracy in preventing specific attack vectors with no false positives. Full pointer-based $\text{Memory Safety}$ is the only truly robust metric.
2. **Low Cost:** Maximum practical runtime overhead is treated as ~10%, with 5% being the industry-preferred threshold. As overhead scales, adoption likelihood drops proportionally:

$$
\text{Adoption likelihood} \downarrow \quad \text{as} \quad \text{runtime overhead} \uparrow
$$

3. **High Compatibility:** The solution must interoperate smoothly with existing binaries and modular compilation workflows.

The defenses that see universal deployment today are comparatively weak, but their overhead is negligible and their compatibility costs are close to zero. That is the core tension that drives everything that follows.

## 1.2 How Stack-Based Buffer Overflow Exploitation Actually Works

The modern memory corruption story starts with a classic primitive: the stack-based buffer overflow. Understanding how it works makes later defenses easier to evaluate, because nearly every mitigation is responding to some part of this original exploit geometry.

### Process Memory and the Stack

A standard process memory layout can be modeled as a composition of three regions:

$$
\text{Process Memory} \approx [T \mid D \mid S]
$$

- **Text ($T$):** The program sequence and read-only data.
- **Data ($D$):** Initialized and uninitialized static variables.
- **Stack ($S$):** The run-time calling frame memory.

The stack operates on a Last-In-First-Out (LIFO) discipline, which is vital for function calls. If an item $x$ is pushed onto stack state $\Sigma$, the immediate top is $x$:

$$
top(push(x,\Sigma)) = x
$$

Because function metadata (like return addresses) is stored here, the Stack ($S$) is the primary battleground for buffer overflows.

### The Geometry of a Stack Frame

On a 32-bit x86 architecture, the stack grows downwards. A stack frame is anchored by two registers: the Stack Pointer (`SP` / `ESP` on x86) tracking the logical top, and the Frame Pointer (`FP` / `EBP`) serving as a stable reference.

Variables are accessed via offsets from the Frame Pointer. Parameters are pushed by the caller and sit at positive offsets, while local variables sit at negative offsets below the Frame Pointer:

$$
addr(\text{param}_i) = FP + \delta_i,\qquad addr(\text{local}_j) = FP - \lambda_j
$$

When a function begins, the standard procedure prolog preserves the previous boundaries, allocating space $L$ for local variables:

$$
[ESP_1] = EBP_0 \quad \text{(Save old FP)}
$$

$$
EBP_1 = ESP_1 \quad \text{(Set new FP)}
$$

$$
ESP_2 = ESP_1 - L \quad \text{(Allocate local storage)}
$$

Consider this minimal function:

```c
void function(int a, int b, int c) {
   char buffer1[5];
   char buffer2[10];
}
```

The compiler allocates frame storage in 4-byte aligned blocks. Thus, `buffer1[5]` takes $8$ bytes and `buffer2[10]` takes $12$ bytes ($L = 20$). The resulting stack layout consists of the local variables sitting directly below the Saved Frame Pointer (`SFP`) and the Return Address (`RET`).

### Turning an Overflow into a Control Primitive

Functions like `strcpy` copy characters indiscriminately until a null byte. The number of overflow bytes can be formulaically defined by the source size $N_{\text{src}}$ and the local buffer capacity $N_{\text{buf}}$:

$$
O = \max(0,\; N_{\text{src}} - N_{\text{buf}})
$$

If an attacker floods a localized buffer with the letter `'A'` (`0x41`), this unbounded byte copy naturally travels up the stack frame, overwriting the Saved Frame Pointer and the Saved Return Address. When the function issues `ret`, the CPU evaluates the corrupted word:

$$
RET_{\text{new}} = 0x41 + 0x41 \cdot 2^8 + 0x41 \cdot 2^{16} + 0x41 \cdot 2^{24} = 0x41414141
$$

The process faults because `0x41414141` is likely an invalid instruction address.

To exploit this properly, an attacker calculates the exact byte distance $\Delta$ from their target buffer to the return address. In our earlier `example1` code, `buffer1` occupies 8 aligned bytes, and the SFP takes 4 bytes.

$$
\Delta_{\text{buffer1}\rightarrow RET} = align_4(5) + 4 = 12
$$

Knowing this fixed offset allows precise control over the return address. An attacker can overwrite it to branch to any instruction, adjusting the jump offset intentionally:

$$
RET_{\text{target}} = RET_{\text{old}} + \Delta_{\text{jump}}
$$

This concept can be translated into direct pointer arithmetic:

<details><summary>Code</summary>

```c
void function(int a, int b, int c) {
   char buffer1[5];
   int *ret;
   // Jump exactly 12 bytes to the saved Return Address
   ret = (int *)(buffer1 + 12);
```

### From Return Address Control to Shellcode Execution

Gaining control of `RET` is step one. To hijack the machine, an attacker returns execution to their own injected byte array, known as shellcode, typically executing something like `execve("/bin/sh")`.

On 32-bit Linux, executing `execve` requires placing exactly the right pointers in the correct registers prior to triggering interrupt `$0x80`:

$$
(EAX, EBX, ECX, EDX) = (11, s, s + 8, s + 12)
$$

Where $s$ is the address of `"/bin/sh"`.

Because the attacker doesn't know the exact absolute memory address of their injected script beforehand, the shellcode must use Position-Independent Code (PIC). A classic trick is to issue a `call` instruction, which natively pushes the address of the next byte to the stack, allowing the payload to `pop` it and dynamically recover its own location:

$$
ESI = RA_{\text{call}} = s
$$

A sample of such handcrafted shellcode (before translation to null-free machine bytes) functions similarly to this assembly block:

<details><summary>Code</summary>

```c
__asm__("
        jmp    0x1f                /* Jump to end to execute call */
        popl   %esi                /* Pop string pointer into ESI */
        movl   %esi,0x8(%esi)      /* Construct argv pointer */
        xorl   %eax,%eax           /* Generate 0 without NULL bytes */
        /* ... Syscall execution omitted for brevity ... */
        int    $0x80               /* execve('/bin/sh') */
        /* ... Exit cleanly ... */
        call   -0x24               /* Call pushes address of the string below */
        .string \"/bin/sh\"
");
```

</details>

To account for variances in memory layouts, attackers frequently pad the space before their sequence with a "NOP sled" - an array of No-Operation bytes. When calculating the payload structure, centering the shellcode ensures a wide buffer zone allows `RET` to land anywhere in the NOP runway safely:

$$
p_{\text{sc}} = buff + \left(\frac{bsize}{2} - \frac{|sc|}{2}\right)
$$

$$
RET_{\text{hit}} \in [buff,\ p_{\text{sc}} - 1] \Longrightarrow \text{eventual execution reaches } sc
$$

### When the Vulnerable Buffer Is Too Small

Large local buffers can hold both the NOP sled and the payload. Small ones cannot. In those cases, attackers separate control hijacking from payload storage.

At startup, a Unix process initializes its stack with arguments ($argv$) and environment variables ($envp$):

$$
\text{Initial Stack} \approx \langle \text{strings} \rangle \; \langle \text{argv pointers} \rangle \; NULL \; \langle \text{envp pointers} \rangle \; NULL \; argc \; argv \; envp
$$

Because environment strings occupy massive contiguous blocks near the top of the stack, the attacker can stage a gigantic `EGG` (the NOP sled plus shellcode) in a dedicated environment variable. The tiny, vulnerable buffer is solely utilized to smash the return address over and over to a highly-probable approximated address:

$$
addr_{\text{guess}} = ESP_{\text{self}} - offset
$$

This leads to robust staged exploitation logic:

<details><summary>Code</summary>

```c
// 1. Allocate massive NOP sled and shellcode into the Egg
ptr = egg;
for (i = 0; i < eggsize - strlen(shellcode) - 1; i++)
  *(ptr++) = NOP;
for (i = 0; i < strlen(shellcode); i++)
  *(ptr++) = shellcode[i];

// 2. Export variables into the environment
memcpy(egg,"EGG=",4); putenv(egg);
memcpy(buff,"RET=",4); putenv(buff); // Filled with repeating addr_guess

// 3. Launch target process utilizing compromised environment footprint
system("/bin/bash");
```

</details>

Ultimately, these exploits succeed because source code performs bounds-agnostic memory writes. At the source level, the key diagnostic condition is simple:

$$
\text{Overflow Risk} \Longleftrightarrow L_{\text{input}} \ge C_{\text{dest}} \;\land\; \text{no effective bound enforcement}
$$

That observation is old, but it still matters. Most later defenses are trying to contain the consequences of exactly this kind of unchecked write.

## 2. Memory-Safety Defenses: CHERI, Capabilities, and Narrower Hardware Mitigations

## 2.1 CHERI and the Capability Approach to Memory Safety

Many defenses try to make exploitation harder after memory-unsafe code has already been written. CHERI takes a different path: change the machine model itself so that authority travels with pointers and can be checked directly by hardware.

### Replacing Plain Pointers with Architectural Capabilities

A traditional processor treats a pointer simply as an integer memory address. CHERI replaces this paradigm with **architectural capabilities** - fat pointers that encapsulate not just location, but explicit, hardware-enforced authority.

A CHERI capability can be logically modeled as a tuple containing an unforgeable validity tag, an address, bounds, permissions, and a sealing state:

$$
c = (tag, addr, base, top, perms, otype)
$$

Hardware checks these attributes transparently at runtime. For any capability-mediated access to succeed, the capability must be valid ($tag=1$), unsealed ($otype=-1$), mathematically in bounds, and carry the semantic permission for the requested operation:

$$
DerefOK(c, op, a) \iff tag(c)=1 \land otype(c)=-1 \land base(c) \le a < top(c) \land op \in perms(c)
$$

To prevent privilege escalation, the ISA natively enforces **capability monotonicity**. Ordinary capability instructions can only shrink bounds and reduce permissions; they can never expand authority:

$$
c' = f(c) \Longrightarrow perms(c') \subseteq perms(c) \ \land \ [base(c'), top(c')) \subseteq [base(c), top(c))
$$

This monotonicity extends to execution states, ensuring that untrusted code cannot increase the authority it can reach without a proper domain transition:

$$
R_{t+1} \subseteq R_t
$$

Even legacy accesses are securely captured under this model via special capabilities, like the Default Data Capability ($DDC$) and Program Counter Capability ($PCC$):

$$
legacy_load(x) \equiv cap_load(DDC, x), \qquad fetch(pc) \equiv cap_fetch(PCC, pc)
$$

Memory address behavior seamlessly adapts based on the process translation regime:

$$
space(c) = \begin{cases} \text{physical or guest-physical}, & \text{if translation is disabled} \\ \text{virtual}, & \text{if translation is enabled} \end{cases}
$$

### Making Capability Metadata Practical in Hardware

Implementing such metadata on real silicon demands care. A naive uncompressed capability footprint is far too expensive (four times native word width, or $4W$), so CHERI actively compresses the bounds structure using floating-point logic down to $2W$:

$$
S_{\text{naive}} = 3W + M \approx 4W, \qquad S_{\text{compressed}} \approx 2W
$$

As allocations grow, bounds compression introduces stronger alignment requirements:

$$
n_1 \le n_2 \Longrightarrow A(n_1) \le A(n_2)
$$

Because real C code temporarily points out-of-bounds without dereferencing, the compressed representation must keep addresses securely representable up to a limit:

$$
[base, top) \subseteq [rep_{lo}, rep_{hi})
$$

$$
DerefOK(a) \implies Representable(a)
$$

Furthermore, maintaining tag coherence without severe latency is achieved using a **Hierarchical Tag Table**. Tag traffic only scales as the specific valid capability pointer density scales across physical pages:

$$
\rho_1 \le \rho_2 \Longrightarrow T_{\text{hier}}(\rho_1) \le T_{\text{hier}}(\rho_2)
$$

### Formal Semantics and Verified Enforcement

CHERI avoids ambiguity by embedding its architecture via formal specifications using the **Sail** language. Instructions rely on a rigid abstract decoding model:

$$
decode(w) = i \quad\Longrightarrow\quad \sigma' = execute(i, \sigma) \ \text{or}\ e = exception(i, \sigma)
$$

This explicit state modeling makes key invariants - such as unauthorized state preservation - precise enough to verify across executions:

$$
\forall x \in P,\ \sigma'(x) = \sigma(x)
$$

When an untrusted component executes inside its controlled isolation boundary, its hardware effects are effectively contained:

$$
ExecWithinBoundary(U, B, \sigma) = \sigma' \Longrightarrow Effects(\sigma,\sigma') \subseteq B
$$

The Sail specification natively incorporates the precise check bounds explicitly rather than utilizing vague prose:

$$
CanLoad(c, a, size) \iff tag(c)=1 \land \neg sealed(c) \land permit_load(c)=1 \land base(c) \le a \land a + size \le top(c) \land aligned(a, size)
$$

Because it acts as the primary hardware simulator, this produces executable instruction conditions like the CHERI `CLoad` operation:

### Software, Revocation, and Compartment Boundaries

In software, memory is reachable only when there is a valid capability path logically connecting root execution registers to the memory object:

$$
Reachable(o) \iff \exists p \in \mathcal{P}(R, o)
$$

For codebases operating under Pure-Capability execution, basic integer addressing is effectively stripped away. The system enforces $DDC = NULL$, throwing exceptions unconditionally if code attempts to reference integer memory improperly:

$$
DDC = NULL \Longrightarrow \forall a,\ \neg IntDerefOK(a)
$$

Temporal memory safety is provided systematically by delaying space reallocation until all capability tags pointing to the freed chunk have been safely swept and revoked:

$$
ReuseAllowed(m) \iff \forall c \in Caps(m),\ \neg DerefOK(c)
$$

Scaling software compartmentalization happens inside single cohesive address spaces rather than utilizing MMU-fragmented IPCs. Compartments enforce capability subsets on shared and closed graph allocations:

$$
\forall c \in C,\ targets(c) \subseteq C \cup S
$$

$$
c' \preceq c \Longrightarrow targets(c') \subseteq targets(c)
$$

### From Research Prototype to Industrial Evaluation

Building CheriBSD proved that CHERI architecture accommodates multiple distinct environment deployments concurrently (`L`: Legacy, `H`: Hybrid, `P`: Pure-Capability):

$$
\mathcal{E} = \{L, H, P\}, \qquad Hostable(k) \subseteq \mathcal{P}(\mathcal{E})
$$

For practical OS function across swapping operations, the kernel reconstitutes bounds through system metadata to preserve logical continuity - forming an abstract capability:

$$
AbstractCap_{t+1} = Rederive(r, m_t)
$$

It also guarantees that system requests touching memory stay within the user-bounded limits supplied during the syscall:

$$
KernelAccess(u, req) \subseteq Targets(c_{arg})
$$

The practical porting effort mathematically reflects the "pointer hygiene" and arbitrary custom runtime operations within individual apps:

$$
PortCost(a) \propto PtrIntConfusion(a) + CustomRuntimeOps(a)
$$

Transforming theoretical capability mechanisms into industrial silicon relies heavily on the **Arm Morello** demonstration architecture. Morello cleanly establishes a comprehensive evaluation baseline merging standard ARM logic dynamically alongside native capability semantics:

$$
MorelloISA = \text{ARMv8-A}_{AArch64} \oplus CHERI_{ISAv8}
$$

Hardware implementations experimentally compare physical tag structures dynamically inside standard memory architectures such as Error-Correcting Code integration mappings:

$$
TagMode \in \{ECC,\ TagController+TagCache\}
$$

Because adoption depends on credible security and overhead measurements, the hardware evaluation pairs CHERI execution with native uninstrumented software:

$$
\mathcal{S}_{run} = \mathcal{S}_{CHERI} \cup \mathcal{S}_{legacy}
$$

Ultimately, making capability security practical requires reducing the total overhead across its major cost centers:

$$
\min \; O_{\text{total}} = O_{\text{spatial}} + O_{\text{referential}} + O_{\text{temporal}} + O_{\text{pointer-dense}}
$$

CHERI matters because it does not merely harden a few exploit edges. It attempts to replace the default authority model of the machine.

## 2.2 Hardware Defenses Short of Full Capability Safety

Not every deployment can adopt a capability architecture. Modern CPUs have therefore added narrower defenses that target distinct layers of the memory corruption problem: region-level access control, pointer integrity, and object-granularity checking.

### Intel Memory Protection Keys (MPK): Region-Level Policy Switching

Traditional page-permission models rely heavily on Translation Lookaside Buffer (TLB) invalidations to revoke or modify access. However, for workloads like cryptographic key management or in-memory databases that require extremely frequent adjustments to access policies, standard `mprotect()` calls suffer from massive latency.

MPK solves this by splitting page classification from capability switching. Up to 16 logical groups of memory pages (keys) are defined in upper page-table entries. However, the exact read/write permissions for those keys are stored cleanly within a thread-local control register (PKRU register).

Because there are $16$ possible keys to designate, and each key gets a write-disable ($WD$) and an access-disable ($AD$) bit, the control register has a capacity boundary of $32$ bits:

$$
|K| = 2^4 = 16, \qquad B = 2|K| = 32
$$

From this configuration, the effective memory access rule for a page $p$ protected by key $k(p)$ running on thread $t$ is naturally resolved purely through this register state:

$$
allow_read_t(p) = \neg AD_t(k(p)), \qquad allow_write_t(p) = \neg AD_t(k(p)) \land \neg WD_t(k(p))
$$

In effect, this allows software to execute large-scale, thread-safe memory isolation state changes immediately using a single non-privileged register arithmetic operation rather than invoking an expensive kernel payload.

### Arm Pointer Authentication (PAC): Pointer-Level Integrity

Where MPK isolates memory chunks, Arm PAC targets the rampant exploitation of control-flow vectors via overwritten pointers such as saved return addresses (RAs) and procedure linkage tables. Standard software-scale checks can be computationally prohibitive on resource-constrained embedded networks. PAC leverages unused bits within modern 64-bit addresses to embed an inline cryptographic signature alongside the pointer data itself.

Instead of hiding the pointer via encryption, PAC authenticates it using a hardware-tuned tweakable block cipher named QARMA. The tag - or Pointer Authentication Code ($PAC$) - is derived from the pointer $p$, a $64$-bit environmental modifier known as the context $c$, and a secret system authentication key $K$:

$$
PAC_n(p, c; K) = Trunc_n\!\left(QARMA_K(p, c)\right)
$$

When `AUT*` verifying instructions invoke PAC validation, they assess if the computed PAC logically equals the tag accompanying the pointer. Crucially, on failure, it doesn't throw an expensive trap. It corrupts the pointer systematically to trigger an ordinary MMU crash upon inevitable dereferencing, dramatically lowering instruction footprint overheads:

$$
AUTH_n(p, \pi, c; K) = \begin{cases} \text{valid pointer } p, & \text{if } \pi = PAC_n(p, c; K) \\ \text{illegal address pattern}, & \text{otherwise} \end{cases}
$$

In the specific case of protecting a stack frame return address, the context chosen is heavily tied to the environment, typically validating the Link Register ($X30$) via the Stack Pointer ($SP$):

$$
PAC^{\text{return}}_n = PAC_n(X30, SP; K_{\text{IA}})
$$

#### Comparing Protections in Assembly

Traditional Software Stack Protection (SSP) inserts bloated verification instructions sequentially, checking stored canaries via multiple operations:

<details><summary>Code</summary>

```asm
/* Old SSP Method: */
SUB sp, sp, #0x50
STP x29, x30, [sp, #0x40]
ADD x29, sp, #0x40
ADRP x3, {pc}
LDR x4, [x3, #SSP]
STR x4, [sp, #0x38]
/* ... executed block ... */
LDR x1, [x3, #SSP]
LDR x2, [sp, #0x38]
CMP x1, x2              /* Manual Verify */
B.NE __stack_chk_fail   /* Error Fallback */
LDP x29, x30, [sp, #0x40]
ADD sp, sp, #0x50
RET
```

</details>

Contrasted directly with PAC's inline cryptographic verification on native architectures utilizing `PACIASP` and `AUTIASP`:

<details><summary>Code</summary>

```asm
/* PAC Execution: */
PACIASP                 /* Inline Key Generation and RA Signing */
SUB sp, sp, #0x40
STP x29, x30, [sp, #0x30]
ADD x29, sp, #0x30
/* ... executed block ... */
LDP x29, x30, [sp, #0x30]
ADD sp, sp, #0x40
AUTIASP                 /* Validate pointer via SP */
RET                     /* Crash inevitably on failure */
```

</details>

While brute force remains technically possible, calculating the accurate tag ($n$-bits) demands exponential guessing. For an attacker to forge an exploit chain over $m$ chained pointers simultaneously, their success probability scales terribly:

$$
\Pr[\text{forge all } m] = 2^{-mn}
$$

### Arm Memory Tagging Extension (MTE): Fine-Grained Memory Defense

Armv8.5-A extends spatial protection aggressively through Memory Tagging Extension (MTE). Often referred to as "Lock-and-Key" architecture, MTE protects memory dynamically at incredible precision. Every single distinct $G = 16$ bytes of granular physical allocation registers a tiny tag lock, and addressing pointers wield a matching top byte key.

Leveraging Top Byte Ignore (TBI) functionalities guarantees negligible metadata overhead at roughly $3.125\%$:

$$
G = 16 \text{ bytes}, \qquad b = 4 \text{ bits}
$$

$$
\text{metadata overhead} = \frac{b}{8G} = \frac{4}{128} = \frac{1}{32} = 3.125\%
$$

Only when the top byte value translates flawlessly with the granular block tracking will data access successfully proceed without generating precise or deterministic kernel fault reporting:

$$
allow(a) = \begin{cases} 1, & \text{if } T_{\text{ptr}}(a) = T_{\text{mem}}(a) \\ 0, & \text{if } T_{\text{ptr}}(a) \neq T_{\text{mem}}(a) \end{cases}
$$

Because tags are strictly $4$-bits long, meaning exactly 16 potential values exist, MTE employs pseudo-random assignment to capture unauthorized wild-pointer overwrites. The mechanism statistically tracks errant behavior where each unique execution carries a $1\text{ in }16$ vulnerability factor. Across scaled test-fuzz deployments iterating reliably across $n$ runs, capturing a violation scales asymptotically toward 100%:

$$
\Pr[\text{detected at least once in } n \text{ runs}] = 1 - \left(\frac{1}{16}\right)^n
$$

More importantly, it provides critical deterministic adjacent detection natively. So long as dynamic allocators strictly issue distinct tags locally across immediately packed structures (e.g., contiguous variables in the same scoped frame), contiguous buffer overflows fault instantaneously at the moment they physically overreach:

$$
T(A_i) \neq T(A_{i+1}) \;\Longrightarrow\; \text{any immediate overflow from } A_i \text{ into } A_{i+1} \text{ mismatches}
$$

MPK, PAC, and MTE do not solve the whole problem, but together they show a common trend: hardware is increasingly taking responsibility for security properties that software alone could not enforce cheaply enough.

## 3. Control-Flow Defenses: CET, CFI, and Hardware-Backed Execution Integrity

## 3.1 Intel CET and the Hardware Enforcement of Control Flow

As attackers shifted from code injection to code reuse, control-flow redirection became a primary target. Intel's **Control-flow Enforcement Technology (CET)** addresses that shift through hardware support for both backward-edge and forward-edge integrity.

At the core of both mechanisms is a dedicated architectural fault class: the Control Protection Exception (`#CP`). CET rarely relies on generic page faults, instead surfacing strict control-flow anomalies natively to the operating system.

### Shadow Stack: Protecting the Backward Edge

Modern ROP attacks work by overwriting saved return addresses on the stack. CET mitigates this by duplicating the return address secretly onto a secondary, hardware-protected "Shadow Stack" that ordinary store instructions cannot access.

The security guarantee hinges on comparing the two domains. On every return, the processor confirms that the address popped from the unprotected data stack identically matches the one safely preserved on the shadow stack. Crucially, the shadow stack memory reads must originate strictly from memory paged exclusively for shadow stacks:

$$
AcceptRET(RA_{\text{data}}, RA_{\text{shadow}}, s) = [RA_{\text{data}} = RA_{\text{shadow}}] \land P_{\text{shadow}}(s)
$$

#### Enabling the Shadow Stack

The feature is intentionally constrained and privilege-aware. The environment must be in protected mode, natively supporting pages, and CET enabled in `CR4`. The architectural test to evaluate if the Shadow Stack remains active at the Current Privilege Level (`CPL`) is modeled formally:

$$
ShadowStackEnabled(c) = [CR4.CET = 1] \land [CR0.PE = 1] \land [EFLAGS.VM = 0] \land SH_STK_EN(c)
$$

In pseudocode, this effectively acts as an execution gate that funnels the checks down to `CPL`-specific machine state registers (`IA32_U_CET` vs `IA32_S_CET`):

<details><summary>Code</summary>

```text
IF CR4.CET = 1 AND CR0.PE = 1 AND EFLAGS.VM = 0
  IF CPL = 3
    SHADOW_STACK_ENABLED = IA32_U_CET.SH_STK_EN;
  ELSE
    SHADOW_STACK_ENABLED = IA32_S_CET.SH_STK_EN;
  FI;
  return SHADOW_STACK_ENABLED == 1;
ELSE
  return FALSE;
ENDIF
```

</details>

Whenever CET is enabled, a standard near `CALL` decrements the hidden architectural Shadow Stack Pointer (`SSP`), copying the return address ($RA$):

$$
SSP' = SSP - w, \qquad Shadow[SSP'] = RA
$$

As returning naturally triggers the architectural evaluation mentioned earlier:

$$
NearRET_{\text{ok}} = [RA_{\text{data}} = RA_{\text{shadow}}]
$$

#### Supervisor Stack Tokens

Things get more complex for supervisor streams across privilege transitions, such as switching from User (`CPL3`) to Supervisor mode. To prevent two logical cores from simultaneously claiming the exact same protected shadow stack (and corrupting the shadow stack metadata), CET implements an ownership lock via a `64`-bit root token ($\tau$). Before `SSP` maps to the new kernel stack address $a$, the token validates its busy bit (bit 0), unreserved status bits, and matching bounds:

$$
ValidSupToken(\tau, a) = [\tau_0 = 0] \land [\tau_{2:1} = 0] \land [\tau_{63:3} = a_{63:3}]
$$

If the conditions pass, the lock is acquired and the pointer advances atomically:

$$
\tau' = \tau \lor 1, \qquad SSP' = a
$$

<details><summary>Code</summary>

```text
1. tau = shadow_stack_load(IA32_PLx_SSP)
2. Require tau.busy == 0 and tau.reserved == 0
3. Require tau.address == IA32_PLx_SSP
4. shadow_stack_store(IA32_PLx_SSP, tau with busy = 1)
5. SSP = IA32_PLx_SSP
```

</details>

Furthermore, instructions like `RSTORSSP` and `SAVEPREVSSP` facilitate context swapping cleanly between concurrent shadow stacks by trading `Restore` ($\rho$) and `PreviousSSP` ($\pi$) tokens, distinguished functionally via their configuration bit ($m$):

$$
\rho_{63:2} = SSP_{\text{saved}}, \qquad \rho_1 = 0, \qquad \rho_0 = m
$$

$$
\pi_{63:2} = SSP_{\text{old}}, \qquad \pi_1 = 1, \qquad \pi_0 = m
$$

### Indirect Branch Tracking (IBT): Protecting the Forward Edge

Where the shadow stack shields return executions, IBT aims to block COP/JOP attacks from redirecting indirect `CALL` or `JMP` operations toward arbitrary memory payload paths.

The approach centers on a tracking state machine dictating compiler-implanted software Landing Pads. Every valid destination of an indirect transaction must be decorated intentionally with an `ENDBR32` or `ENDBR64` marker. Thus, whenever an indirect jump occurs, the CPU alters its tracker loop ($S_{i}$), requiring the following immediate validation context:

```math
S_{i+1} =
\begin{cases}
S_{\mathrm{wait}}, & \text{if } I_i \in \{\text{indirect CALL}, \text{indirect JMP}\}, \\
S_{\mathrm{idle}}, & \text{if } S_i = S_{\mathrm{wait}} \land I_{i+1} = \text{ENDBRANCH}, \\
\text{\#CP}, & \text{if } S_i = S_{\mathrm{wait}} \land I_{i+1} \neq \text{ENDBRANCH}.
\end{cases}
```

The beauty of `ENDBR32/64` lies in backward compatibility. On older x86 systems, the encoding safely executes as a functional `NOP`, which allows a single binary distribution to remain usable across CET and non-CET environments.

#### Suppressions and Target Escapes

Actively checking for an `ENDBRANCH` happens conditionally based on suppression states. A tracker remains fully armed strictly when it is enabled globally and not temporarily downgraded via a suppression bit:

$$
TrackActive(c) = EndbranchEnabled(c) \land [SUPPRESS(c) = 0]
$$

<details><summary>Code</summary>

```text
IF CR4.CET = 1 AND CR0.PE = 1 AND EFLAGS.VM = 0
  IF CPL = 3
    ... (fetch U_CET EN/SUPPRESS rules)
  ELSE
    ... (fetch S_CET EN/SUPPRESS rules)
  FI;
  return (ENDBR_ENABLED == 1) AND (SUPPRESSED == 0);
ELSE
  return FALSE;
ENDIF
```

</details>

Not every transaction triggers validation unconditionally. To handle intricate custom compilations, a designated instruction prefix (`3EH`) completely bypasses the sequence for standard near tracking logic when configured:

$$
TrackedNear(b) = IndirectNear(b) \land \left(\neg Prefix_{3E}(b) \lor [NO_TRACK_EN = 0]\right) \land TrackActive(CPL)
$$

#### Legacy Compatibility Modes

Since transitioning entire system codebases (including legacy shared objects) is demanding, CET affords targeted exemptions based on legacy bitmaps defining allowed memory pages. During a potential `#CP` violation, an indexed resolution targets structural memory chunks logically formatted using address indexing formats to establish approval arrays:

$$
byte_index = LA[\alpha:15], \qquad bit_index = LA[14:12]
$$

$$
LegacyAllow(LA) = Bitmap[byte_index]_{bit_index}
$$

If the target domain lacks the required `ENDBRANCH` landing pad but inherently passes the bitmap lookup resolution matrix, the IBT machine quietly overrides the fault and functionally relaxes the state tracker down dynamically:

<details><summary>Code</summary>

```text
IF LEG_IW_EN == 1
  LA = target linear address
  BITMAP_BYTE = load(bitmap entry selected by LA)
  IF bitmap bit for target page == 0
    Deliver #CP(ENDBRANCH)
  ELSE
    TRACKER = IDLE
    SUPPRESS = (SUPPRESS_DIS == 0) ? 1 : 0
    Restart the instruction
  ENDIF
ELSE
  Deliver #CP(ENDBRANCH)
ENDIF
```

</details>

By combining backward-edge duplication with forward-edge landing-pad validation, CET narrows one of the most important exploit surfaces in modern systems.

## 3.2 Control-Flow Integrity as the Software Policy Foundation

If CET is a hardware realization of specific control-flow guarantees, **Control-Flow Integrity (CFI)** is the broader software policy that defines what those guarantees should mean. CFI is important because it changes the question from "did something suspicious happen?" to "did execution follow a permitted edge in the program's control-flow graph?"

### The Core Policy: Execution Must Stay on the Graph

The fundamental premise of CFI is that a program's execution trace $\pi$ should always be constrained to a predetermined, mathematically defined Control-Flow Graph (CFG) $G = (V,E)$. Every subsequent step in the execution cycle must adhere strictly to the static edges identified during instrumentation:

$$
CFI(\pi, G) \iff \forall i \in \{0, \ldots, n-1\},\ (v_i, v_{i+1}) \in E
$$

Unlike probabilistic random-bit-flip fault tolerance systems ($\Pr[\text{deviation}] \le \varepsilon_{\text{soft}}$), CFI presents a stringent security boundary designed around a massively empowered threat model. Even if an attacker possesses the ability to randomly trample and alter pure data memory states dynamically ($A_t$), the architectural guardrails dictate that the instruction target $pc_{t+1}$ remains safely enclosed within valid subsequent nodes:

$$
\forall t,\ \forall \mathcal{A}_t,\ M_{\text{data}}^{(t+1)} = \mathcal{A}_t\!\left(M_{\text{data}}^{(t)}\right) \Longrightarrow pc_{t+1} \in Succ_G(pc_t)
$$

### Inlined Enforcement in Real Binaries

To move from theory into practice without specialized hardware, CFI utilizes **inlined enforcement**. Indirect jumps and dynamic transfers are the primary vulnerabilities, necessitating dynamic source-side ID checks matched logically against destination tags.

Ideally, two destinations share a grouped tag merely if they share exactly the identical source domains:

$$
d_1 \sim_G d_2 \iff Pred_G(d_1) = Pred_G(d_2)
$$

When compiled, the binary rewriter injects semantic ID labels immediately accessible at the destinations. Meanwhile, check routines validating $Accept(s,d)$ enforce that the dynamically derived call targets natively align with their expected IDs ($I_s$):

$$
Accept(s,d) \iff ID(d) = I_s
$$

#### x86 Structural Instrumentation

Practically translating this on variable-length legacy x86 architectures introduces options surrounding how these IDs populate memory and mitigate Time-of-Check-to-Time-of-Use (TOCTTOU) constraints. Consider the two alternative implementations below:

<details><summary>Code</summary>

```asm
; Alternative (a) ID stored as raw data before the destination
; Alters the final pointer to skip raw ID bytes
cmp [ecx], 12345678h
jne error_label
lea ecx, [ecx+4]
jmp ecx
dd 12345678h      ; raw ID bytes
mov eax, [esp+4]

; Alternative (b) Effect-free label via prefetchnta
; Prefetchnta creates an inert architectural label directly
mov eax, 12345677h
inc eax
cmp [ecx+4], eax
jne error_label
jmp ecx
prefetchnta [12345678h]
mov eax, [esp+4]
```

</details>

### Real-World Approximations and Their Cost

Producing the valid bounds requires reconstructing the CFG via binary analysis. Because resolving arbitrary pointer behavior precisely is famously undecidable, the implementation embraces conservative approximations. At worst, any function holding a relocation or taken address may be permitted:

$$
Targets_{\text{true}}(c) \subseteq Targets_{\text{impl}}(c) = \{\, f \mid AddrTaken(f) \,\}
$$

If any single step of the exploit necessitates jumping illegally (such as jumping into the unaligned middle of an instruction stack) - the system throws a hard termination:

$$
\exists t \text{ such that } (pc_t, pc_{t+1}) \notin E \Longrightarrow \text{exploit is stopped at step } t
$$

The performance impact is measured relative to the uninstrumented baseline and averages roughly $16\%$ across SPEC benchmarks:

$$
Overhead(P) = \frac{T_{\text{CFI}}(P) - T_{\text{orig}}(P)}{T_{\text{orig}}(P)} \times 100\%
$$

### Building on CFI: SFI, SMAC, and Shadow Call Stacks

Once a rigorous CFG lock acts as the program substrate, cascading systems like Software Fault Isolation (SFI) and Software Memory Access Control (SMAC) scale dramatically more efficiently. CFI explicitly guarantees checks are structurally non-circumventable across security boundaries:

$$
v_n = a \Rightarrow \exists k < n \text{ such that } v_k = Check(a)
$$

For example, memory arrays bounds checking loop bodies no longer suffer masking penalties iteratively because CFI implicitly forbids attackers natively circumventing initialization. Furthermore, SMAC limits control structures deterministically via `AND` restrictions (top byte truncation):

$$
Mask_{40}(x) = x \land \texttt{0x40FFFFFF}
$$

$$
x' = Mask_{40}(x) \Rightarrow TopByte(x') \in \{\texttt{0x00}, \texttt{0x40}\}
$$

#### Protected Shadow Call Stacks

Standard coarse CFI guarantees return constraints but lacks historical contextualization. By incorporating an isolated segment boundary (such as the `gs` register), call/return topologies enforce purely isolated FIFO chains matching calls directly to targeted labels:

$$
Call(L_{\text{ret}}, S) \mapsto S' = S \mathbin{\|} L_{\text{ret}}
$$

$$
Ret(S) \mapsto \bigl(top(S), pop(S)\bigr)
$$

<details><summary>Code</summary>

```asm
; Isolated Push into Segment (Call Phase)
add gs:[0h], 4h
mov ecx, gs:[0h]
mov gs:[ecx], LRET
cmp [eax+4], ID
jne error_label
call eax
LRET: ...

; Isolated Pop from Segment (Return Phase)
mov ecx, gs:[0h]
mov ecx, gs:[ecx]
sub gs:[0h], 4h
add esp, 4h    ; Skips typical unsecure standard frame pointer return
jmp ecx
```

</details>

### Formal Verification and Why CFI Matters

CFI is robust not simply because its mitigations seem plausible, but because of formalized mathematically demonstrated proofs. Abstracting the state definition strictly into programmatic domains:

$$
S = (M_c \mid M_d,\ R,\ pc)
$$

By uniting standard operational execution ($\to_n$) combined structurally with omnipotent adversary capability manipulation constraints ($\to_a$), the comprehensive functional flow emerges:

$$
\to\ =\ \to_n\ \cup\ \to_a
$$

Assuming a well instrumented code memory validation ($I(M_c)$ is verified), the seminal CFI Proof concludes that any step executes valid instruction propagation ($\text{Succ}_{CFG}$), or harmless attacker data manipulation leaving the architectural control frame frozen:

$$
\forall i \in \{0,\ldots,n-1\},\ \bigl(S_i \to_a S_{i+1}\bigr)\ \lor\ \bigl(pc(S_{i+1}) \in Succ_{CFG}(pc(S_i))\bigr)
$$

That result captures the core value of CFI. It does not claim the attacker cannot corrupt data. It claims that even with powerful data corruption capabilities, the attacker should not be able to drive execution off the intended graph.

---

The history of memory corruption defense is not a clean progression from weak ideas to perfect ones. It is a record of tradeoffs. Stack cookies, DEP, and ASLR remain widely deployed because they are cheap. CFI and CET raise the bar against control-flow hijacking because they target one of the most durable exploitation patterns. MPK, PAC, and MTE push more security responsibility into hardware without demanding a total rewrite of existing ecosystems. CHERI goes further by rethinking the authority model of the machine itself.

Taken together, these systems point to the same conclusion: memory safety is not just a research problem in detection or mitigation. It is an engineering problem in deployability. The strongest defense is valuable only if its protection, cost, and compatibility line up well enough for real software stacks to adopt it.
